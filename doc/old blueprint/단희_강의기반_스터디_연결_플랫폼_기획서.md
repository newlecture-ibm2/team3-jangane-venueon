# [서비스 기획서] 강의/세미나 기반 학습 및 스터디 연결 플랫폼

## 1. 서비스 한 줄 정의
**"강의 수강 완료 후, 동일한 주제로 묶인 참여자들과 즉시 후속 스터디를 이어가는 곳"**
오프라인 행사의 현장 체크인(QR)부터 온라인 강의 참석(Zoom 링크 연결)까지 수강 경험을 원활히 지원하고, **수강 이후의 학습 활동을 커뮤니티(스터디 룸)로 지속시켜주는 연결 플랫폼**입니다.

## 2. 문제 정의 (As-Is)
이 서비스가 해결하고자 하는 주요 문제점은 다음과 같습니다.
- **휘발성 짙은 학습 경험**: 강의를 듣고 끝나는 경우가 많아, 실제 학습 행동이나 실무 적용으로 이어지지 않습니다.
- **네트워킹 구조의 부재**: 같은 강의를 들으며 비슷한 관심사를 가진 사람들과 연결될 시스템적 장치가 없습니다.
- **파편화된 사용자 경험**: 사용자가 강의 사전 신청, 온라인(Zoom) 수강, 후속 학습(스터디, 자료 공유)을 각기 다른 플랫폼에서 분산 관리해야 하므로 피로도가 높습니다.
- **운영의 비효율성**: 운영자 입장에서 강의 등록/결제, 참가자 관리, 화상 세션 운영, 후속 커뮤니티 관리가 분리되어 있어 연속적인 브랜딩과 통제가 어렵습니다.

## 3. 해결 방향 (To-Be)
본 플랫폼은 **강의 참석(오프라인 QR 티켓 발급, 온라인 Zoom 접속 링크 제공)을 지원하고, 이벤트 종료 후 동기 부여가 가장 높은 타이밍에 공통 관심사를 가진 스터디 방으로 참여자들을 연결**해 문제를 해결합니다.
- **강의 경험에서 스터디로의 매끄러운 연결(Hub)**: 플랫폼 내 자체적인 영상 송출 통신 기능 등은 과감히 생략합니다. 단, 수강생이 플랫폼에서 부여받은 QR 티켓(오프라인)이나 접속 링크(온라인)를 바탕으로 편하게 강의를 듣고 난 직후, 플랫폼 내에 미리 준비된 스터디 룸으로 바로 진입하여 끊김 없는 후속 학습을 이어갈 수 있도록 허브 역할을 합니다.
- 단순 이벤트 등록을 넘어, 강의 수강 이후 공통 관심사를 가진 수강생들이 스터디 방으로 묶이는 **자연스러운 연결 흐름**을 제공합니다.
- 단순한 '자유 게시판'이 아닌, 질문/정리/토론 등 목적이 명확한 **구조화된 스터디 피드**를 제공합니다.
- 스터디 내 자료 아카이빙 및 일정(다음 스터디 모임) 관리를 결합하여, 사용자들이 목적을 잃지 않고 **자기주도적인 후속 학습**을 지속하도록 돕습니다.

## 4. 사용자 유형 정의

### 학습자 (Attendee / Learner)
- **목표**: 관심 있는 강의/세미나에 참여해 지식을 습득하고, 이후 같은 과정을 들은 사람들과 스터디를 통해 지식을 고도화하며 네트워킹을 쌓습니다.
- **행동 패턴**: 관심 강의 탐색 및 결제 → 온/오프라인 강의 수강(플랫폼 접속 정보 안내) → 스터디 방 참여 후 템플릿 기반 자기소개 작성 → 스터디 피드에 질문/정리/토론 글 작성 → 다음 스터디 모임 일정 확인 및 참여

### 운영자 (Organizer / Instructor)
- **목표**: 효과적으로 자신의 강의를 홍보하고 안정적으로 수강생을 모집·관리하며, 향후 스터디 조성을 통해 팬덤이나 실전 커뮤니티로 발전시킵니다.
- **행동 패턴**: 강의 정보 입력 및 개설 → 결제 내역 및 참가자 명단 관리 → 온/오프라인 강의 진행상태 관리(접속 장소 안내 및 현장 QR 체크인) → 강의 종료 시 후속 스터디 개설 및 기초 학습 자료 업로드

## 5. 핵심 사용자 플로우

### 학습자 플로우
1. **강의 발견**: 피드 및 카테고리별 강의 목록에서 관심 세미나 탐색 및 상세 확인
2. **신청/결제**: 수강 신청 후 PG 모듈을 통한 결제 진행 (내 강의 목록에 추가)
3. **강의 수강 준비/수강**: 플랫폼에서 제공된 정보에 따라 강의 참석 (온라인: 제공된 화상 플랫폼 링크로 이동, 오프라인: 발급받은 QR 티켓으로 현장 체크인 참석)
4. **수강 완료 및 스터디 진입**: 강의 후 개설된 '해당 강의 전용 스터디 방' 입장
5. **온보딩 설정**: 첫 진입 시 모달창에서 자기소개(이름/닉네임, 관심사 태그, 한 줄 소개, 이번 강의에서 궁금한 점) 폼 제출
6. **스터디 참여 및 활동**: 
   - **홈**: 오늘 무엇을 해야 하는지(다음 할 일 카드, 추천 액션 등) 확인
   - **피드**: 템플릿을 활용해 첫 질문/정리 글 작성
   - **자료/일정**: 강의 복습 자료 열람 및 다음 동료 모임 참석 투표

### 운영자 플로우
1. **강의 생성**: 제목, 상세 커리큘럼, 일정, 참가 비용 등 등록
2. **참가자 모집 및 관리**: 수강생 목록 및 결제 상태 모니터링
3. **강의 참석 현황 운영**: 수강생용 안내 정보(스트리밍 접속 링크/장소 등) 사전 등록 및 오프라인 참석자 대상 현장 QR 체크인 확인
4. **스터디 연결 지원**: 스터디 기능 활성화 후 첫 주 안내 방식 고정글(공지) 등록, 강의 슬라이드 공유

## 7. 차별화 포인트
가장 큰 엣지(Edge)는 기존 이벤트 관리 플랫폼(예: Eventbrite 등)과 달리, 단순 티켓팅이나 웨비나 위주에서 끝나는 것이 아닌 **"강의 이후 스터디 연결 생태계"**를 내재화했다는 점입니다. 

자유 게시판을 두면 금방 생명력을 잃고 방치되는 커뮤니티의 단점을 보완하기 위해:
- 철저한 **템플릿(질문, 정리, 토론) 중심 글쓰기**를 제공하여 목적성을 가시화합니다.
- 대시보드 성격의 **강력한 스터디 '홈' 탭**을 통해 사용자가 접속했을 때 즉시 해야 할 행동을 제안합니다. (예: 자기소개를 작성하세요, 이번 주 토론 주제 등)
- 일정 관리와 자료실이 함께 세팅되어, 학습 행동이 외부(단톡방, 외부 클라우드 등)로 유출되지 않고 플랫폼 내에서 지속 순환하는 구조를 가집니다.

## 8. 데이터 구조 (Entity)
데이터베이스 아키텍처 확장에 대비한 핵심 Entity는 다음과 같습니다.

1. **User (사용자)**: 시스템 통합 식별자, 기본 프로필 정보
2. **Course / Seminar (강의/세미나)**: 강의 기본 정보 (제목, 상세설명, 일시, 가격 등)
3. **Registration (수강신청)**: User ↔ Course 간 다대다 연결을 처리하는 테이블 (등록 상태, 결제 식별자 연결)
4. **Payment (결제)**: 결제 금액, 트랜잭션 아이디, 결제수단 및 상태 관리
5. **Ticket & Pass (참석 관리)**: 유저의 참석 방식을 정의하는 데이터 (온라인=송출 링크 관리, 오프라인=발생된 QR 티켓 고유값 및 체크인 상태 확인)
6. **StudyGroup (스터디 그룹)**: Course에 1:1 대응하여 개설되는 후속 스터디 존 메타데이터 (기간, 스터디 현황값)
7. **StudyMembership (스터디 멤버십)**: StudyGroup 내 해당 User의 고유한 스터디 프로필 (활동 한 줄 소개, 관심사 태그, 현 궁금증) 및 액티비티 레벨(글쓴 횟수 등)

## 9. Cloud Native 아키텍처 개요
빠른 배포와 도메인별 트래픽의 독립적 스케일링을 위해 MSA(Microservices Architecture)로 분리합니다.

- **API Gateway**: 클라이언트와 백엔드 간 통신 입구 (라우팅, Auth 토큰 검증)
- **User Service**: 회원가입, 인증, 전체 프로필 정보
- **Course Service**: 강의 개시, 목록/상세 조회, 카테고리 처리 (Read heavy)
- **Registration & Ticket Service**: 수강 신청 정합성 처리, 외부 PG 연동, 온/오프라인 출석 권한(QR 티켓/접속 링크) 생성
- **Study Service**: 피드 글쓰기, 멤버십 프로필 업데이트, 자료 보관함 등 스터디 룸 전반 관리 (Write/Read 빈번, NoSQL 도입 또는 분산 캐시 처리 고려 가능)
- **Notification Service**: 스터디 모임 D-Day 알림, 댓글 달림 등을 위한 비동기 발송 서버
*(이외 외부 결제 모듈, DB, S3 Storage 등 개별 연결 모듈 구성 포함)*

## 10. 주요 API 예시
- `GET /courses` : 진행/예정 중인 강의 목록 리스트 필터링
- `POST /courses` : (운영자 권한) 새 강의 개설
- `POST /courses/{courseId}/registrations` : 수강 상세 페이지에서 신청 및 사전 대기(또는 결제 요청 분기)
- `POST /payments` : 외부 PG 연동 완료 데이터 기반 결제 승인 데이터 생성
- `GET /courses/{courseId}/access` : (학습자) 결제 완료 후 온라인 스트리밍 참석 링크 확인 또는 현장 QR 티켓 조회
- `POST /courses/{courseId}/tickets/check-in` : (운영자) 오프라인 행사 당일 현장 QR 코드 스캔 및 출석 승인
- `POST /courses/{courseId}/studies` : 강의 종료 즈음 후속 스터디 그룹 초기 생성
- `GET /study-groups/{studyId}/feed` : 피드 최신 글, 토론/질문 카테고리 필터링 조회
- `POST /study-groups/{studyId}/feed` : 사용자가 템플릿(정리, 토론 등) 폼 전송
- `POST /study-groups/{studyId}/members/{userId}/onboarding` : 스터디 첫 진입 시 모달에서 자기소개 데이터 전송

## 11. 기술 스택 제안

- **Frontend**: React (단일 페이지 구현 용이), Tailwind CSS (컴포넌트 작성을 위한 CSS 직관성), React Query 혹은 Zustand (서버 상태 및 클라이언트 글로벌 상태)
- **Backend (API 서버)**: Spring Boot (Java) - 객체 지향 및 아키텍처 설계 연습에 용이 
- **Database**: 
  - **RDBMS**: MySQL 또는 PostgreSQL (복잡하지 않은 스키마 설계와 결제 정합성에 적합)
- **Infra (Cloud 연계)**: AWS EC2 배포 기반 (점진적으로 Docker Containerize 하여 Github Actions를 활용한 CI/CD 파이프라인 구성 시도)
- **External API & Service**: 
  - 티켓팅 관리: QR코드 생성 라이브러리(QRCode.js 등) 및 카메라 스캐닝 모듈 활용
  - 결제: 포트원 (PortOne, 구 아임포트) 간편 결제
  - 이미지 처리: AWS S3 저장
