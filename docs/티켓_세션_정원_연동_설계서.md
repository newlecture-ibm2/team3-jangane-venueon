# 티켓-세션 정원 연동 및 모집 상태 설계서

> **Jira**: VO-828 (EC-022 이벤트 세션/티켓 워크플로우)
> **작성일**: 2026-04-11
> **브랜치**: `young/feat/VO-828/EC-022-event-session-ticket`

---

## 1. 개요

티켓 판매 시 연결된 세션의 정원을 자동으로 차감하고,
세션 정원이 가득 차면 해당 세션을 포함하는 티켓을 자동 마감하는 **양방향 정원 연동 시스템**을 설계한다.

> [!IMPORTANT]
> **핵심 원칙: 이중 게이트(Dual Gate)**
>
> 티켓 구매 가능 여부는 두 가지 조건을 **모두** 만족해야 한다:
> 1. **티켓 자체 재고**: `remainingQuantity > 0` (또는 무제한)
> 2. **연결된 세션 정원**: 포함된 모든 세션의 `currentAttendees < maxAttendees`
>
> 어느 한쪽이라도 가득 차면 해당 티켓은 **구매 불가** 상태가 된다.

## 2. 정원 관리 모델

### 2-1. 두 가지 정원의 역할

| 구분 | 주체 | 필드 | 역할 |
|---|---|---|---|
| **티켓 재고** | Ticket | `maxQuantity`, `soldCount` | 이 티켓이 최대 몇 장 팔릴 수 있는지 (호스트 설정) |
| **세션 정원** | Session | `maxAttendees`, `currentAttendees` | 이 세션에 최대 몇 명이 참여할 수 있는지 (호스트 설정) |

> [!NOTE]
> `maxQuantity = null` → 티켓 재고 무제한 (세션 정원에만 제한됨)
> `maxAttendees = 0` → 세션 정원 무제한 (티켓 재고에만 제한됨)

### 2-2. 관계 다이어그램

```
  티켓 A (전체 세션)          티켓 B (개별: 세션1만)
  maxQuantity: 100            maxQuantity: 50
       │                           │
       ├──── 세션 1 ◄──────────────┘
       │     maxAttendees: 80
       │     currentAttendees: ?
       │
       └──── 세션 2
             maxAttendees: 60
             currentAttendees: ?
```

- 티켓 A가 1장 팔리면 → 세션1 +1, 세션2 +1
- 티켓 B가 1장 팔리면 → 세션1 +1만
- 세션1의 `currentAttendees`가 80에 도달하면 → 티켓 A, B **모두 구매 불가**
- 세션2의 `currentAttendees`가 60에 도달하면 → 티켓 A만 구매 불가, 티켓 B는 유지

### 3-3. 티켓 구매 가능 여부 판단 (Computed)

```java
/**
 * 티켓의 실시간 구매 가능 여부를 계산한다.
 * 티켓 재고 + 연결된 모든 세션의 정원 + 모집 상태를 종합 판단
 */
public boolean isTicketPurchasable(Ticket ticket, List<Session> linkedSessions) {
    // 1. 티켓 비활성
    if (!ticket.isActive()) return false;
    
    // 2. 티켓 재고 소진
    if (ticket.getMaxQuantity() != null && ticket.getRemainingQuantity() <= 0) {
        return false;
    }
    
    // 3. 연결된 세션 중 하나라도 정원 초과
    for (Session session : linkedSessions) {
        if (session.getMaxAttendees() > 0 
            && session.getCurrentAttendees() >= session.getMaxAttendees()) {
            return false;
        }
    }
    
    // 4. 연결된 세션 중 하나라도 모집 비활성
    for (Session session : linkedSessions) {
        if (session.getRecruitmentStatus() != RecruitmentStatus.OPEN) {
            return false;
        }
    }
    
    return true;
}
```

### 3-4. 연결된 세션 조회 헬퍼

```java
private List<Session> getLinkedSessions(Ticket ticket) {
    if (ticket.isAllSessions()) {
        // 전체 세션 티켓 → 이벤트의 모든 세션 반환
        return sessionRepository.findAllByEventId(ticket.getEventId());
    } else {
        // 개별 세션 티켓 → ticket_sessions 매핑 테이블에서 조회
        return sessionRepository.findAllByIds(ticket.getSessionIds());
    }
}
```

## 4. 마감 판단 우선순위

티켓이 구매 불가능한 이유를 사용자에게 정확히 전달하기 위한 판단 흐름:

```
티켓 구매 요청
  │
  ├─ 티켓 비활성 (isActive=false)
  │   → "판매 종료된 티켓입니다"
  │
  ├─ 티켓 재고 소진 (remainingQuantity <= 0)
  │   → "매진되었습니다"
  │
  ├─ 세션 정원 초과 (currentAttendees >= maxAttendees)
  │   → "세션 'OOO'의 정원이 마감되었습니다"
  │
  ├─ 세션 모집 기간 외 
  │   ├─ PENDING → "아직 모집이 시작되지 않았습니다 (시작: 4/1)"
  │   └─ CLOSED  → "모집이 마감되었습니다"
  │
  └─ 모두 통과 → 구매 가능 ✅
```

기존 가격/일정/유형 그리드에 **모집** 항목 추가:

```
┌─ 이벤트 정보 ──────────────────────────────────┐
│  가격     ₩15,000 ~ ₩30,000                    │
│  일정     2026.04.20 ~ 2026.04.22              │

│  유형     CONFERENCE                            │

- `모집 시작일`: `min(세션들의 recruitStartDate)`
- `모집 마감일`: `max(세션들의 recruitEndDate)`
- 모집 상태 뱃지: 이벤트의 `recruitmentStatus` (세션 OR 종합)

┌─────────────────────────────────────────────────┐

│  올데이 패스                                     │

│  📋 4/1 ~ 4/19 구매 가능                        │  ← 모집 기간

#### 상태별 표시 변화

| 상태 | 뱃지 | 마감일 표시 | 카드 스타일 |
|---|---|---|---|
| 구매 가능 (OPEN) | 🟢 `구매가능` | `📋 4/1 ~ 4/19 구매 가능` | 기본 |
| 마감 임박 (D-3 이내) | 🟠 `마감임박` | `⚠️ D-2 마감임박! (4/19까지)` | 주황 강조 |
| 모집 대기 (PENDING) | ⏳ `모집대기` | `📋 4/1부터 구매 가능` | 회색 처리 |
| 정원 마감 | 🔴 `정원마감` | `세션 'OOO'의 정원이 마감되었습니다` | 비활성 |
| 티켓 매진 | 🔴 `매진` | — | 비활성 |
| 모집 종료 | 🔴 `마감` | `모집이 종료되었습니다` | 비활성 |

#### 마감 임박 판단 기준

```typescript
function getUrgencyLevel(recruitEndDate: string | null): 'normal' | 'urgent' | 'closed' {
  if (!recruitEndDate) return 'normal'; // 마감일 미설정 → 상시 모집
  
  const end = new Date(recruitEndDate);
  const now = new Date();
  const daysLeft = Math.ceil((end.getTime() - now.getTime()) / (1000 * 60 * 60 * 24));
  
  if (daysLeft < 0) return 'closed';
  if (daysLeft <= 3) return 'urgent';  // D-3 이내
  return 'normal';
}
```

## 7. 동시성 처리

> [!CAUTION]
> 인기 이벤트에서 동시에 여러 사용자가 구매를 시도할 때 **정원 초과 판매(Overselling)**를 방지해야 한다.

### 7-1. 비관적 잠금 (Pessimistic Locking)

```java
// SessionRepository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT s FROM EventSessionJpaEntity s WHERE s.id = :id")
Optional<EventSessionJpaEntity> findByIdForUpdate(@Param("id") Long id);

// TicketRepository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT t FROM TicketJpaEntity t WHERE t.id = :id")
Optional<TicketJpaEntity> findByIdForUpdate(@Param("id") Long id);
```

### 7-2. 트랜잭션 흐름

> [!NOTE]
> 아래 트랜잭션 흐름은 **OrderService**에서 구현합니다.
> 상세 구현은 [주문_장바구니_티켓연동_요구사항.md](./주문_장바구니_티켓연동_요구사항.md) 를 참고하세요.
> Event 도메인에서는 비관적 잠금 쿼리(7-1)만 제공합니다.

```
BEGIN TRANSACTION
  │
  ├─ SELECT ticket FOR UPDATE (비관적 잠금)
  ├─ SELECT sessions FOR UPDATE (비관적 잠금)
  │
  ├─ 재고 검증 → 실패 시 ROLLBACK + "매진" 응답
  ├─ 정원 검증 → 실패 시 ROLLBACK + "정원 초과" 응답
  │
  ├─ ticket.soldCount += quantity
  ├─ session.currentAttendees += quantity (각 세션)
  ├─ INSERT order
  │
COMMIT
```

### 7-3. 향후 확장: Redis 기반 재고 관리

MVP 이후 트래픽이 증가하면 Redis를 활용한 재고 캐싱을 고려:

```
Redis: ticket:{id}:stock = 88
Redis: session:{id}:capacity = 68

DECR ticket:{id}:stock → 87 (원자적 연산)
DECR session:{id}:capacity → 67

실패 시 INCR로 복구 (보상 트랜잭션)
DB 동기화는 비동기로 처리
```

## 8. 시나리오별 동작

### 시나리오 1: 정상 구매

```
사전 상태:
  티켓 A (전체 세션) - 재고 100, 판매 0
  세션 1 - 정원 80, 참석 0
  세션 2 - 정원 60, 참석 0

사용자가 티켓 A를 2매 구매:
  → 티켓 A: 판매 2, 잔여 98
  → 세션 1: 참석 2, 잔여 78
  → 세션 2: 참석 2, 잔여 58
```

### 시나리오 2: 세션 정원으로 인한 자동 마감

```
사전 상태:
  티켓 A (전체: 세션1+세션2) - 재고 100, 판매 55
  티켓 B (개별: 세션1만) - 재고 50, 판매 20
  세션 1 - 정원 80, 참석 75 (잔여 5)   ← 거의 가득 참
  세션 2 - 정원 60, 참석 55

사용자가 티켓 B를 6매 구매 시도:
  → 세션 1 정원 검증 실패 (75 + 6 > 80)
  → "세션 'Day 1'의 정원이 부족합니다 (잔여 5석)"

사용자가 티켓 B를 5매 구매:
  → 세션 1: 참석 80, 잔여 0  ← 가득 참!
  → 티켓 A: isPurchasable = false (세션1 정원 초과)
  → 티켓 B: isPurchasable = false (세션1 정원 초과)
  → 두 티켓 모두 자동으로 "정원마감" 상태
```

### 시나리오 3: 티켓 재고로 인한 마감

```
사전 상태:
  티켓 B (개별: 세션1만) - 재고 50, 판매 49
  세션 1 - 정원 80, 참석 69 (잔여 11)

사용자가 티켓 B를 2매 구매 시도:
  → 티켓 재고 검증 실패 (49 + 2 > 50)
  → "티켓 재고가 부족합니다 (잔여 1매)"

사용자가 티켓 B를 1매 구매:
  → 티켓 B: 판매 50, 잔여 0 ← 매진!
  → 세션 1: 참석 70, 잔여 10 (아직 여유)
  → 티켓 B만 "매진", 티켓 A는 세션1 정원이 남아있으면 계속 판매 가능
```

### 시나리오 4: 주문 취소 후 복구

```
사전 상태:
  티켓 A - 판매 60, 잔여 40
  세션 1 - 참석 80, 잔여 0 ← 가득 참 (티켓 A, B 모두 마감)

사용자가 주문(티켓 A, 2매) 취소:
  → 티켓 A: 판매 58, 잔여 42
  → 세션 1: 참석 78, 잔여 2
  → 세션 2: 참석 58, 잔여 2
  → 티켓 A: isPurchasable = true ← 자동 복구!
  → 티켓 B: isPurchasable = true ← 자동 복구!
```
