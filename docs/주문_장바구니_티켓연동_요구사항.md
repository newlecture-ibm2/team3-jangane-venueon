## 1. 개요

티켓 판매(주문 생성) 시 연결된 세션의 정원을 차감하고,
주문 취소 시 재고/정원을 복구하는 로직은 **OrderService** 에서 구현해야 합니다.

핵심 원칙 — **이중 게이트(Dual Gate)**: 티켓 구매 가능 여부는 아래 두 가지를 **모두** 만족해야 합니다:
1. **티켓 자체 재고**: `remainingQuantity > 0` (또는 무제한)
2. **연결된 세션 정원**: 포함된 모든 세션의 `currentAttendees < maxAttendees`

## 2. OrderService — 주문 생성 로직

> 원본 설계서 Section 3-1에서 발췌

```java
@Transactional
public Order createOrder(Long ticketId, Long userId, int quantity) {
    Ticket ticket = ticketRepository.findById(ticketId);
    
    // Gate 1: 티켓 재고 검증
    if (!ticket.hasStock(quantity)) {
        throw new BusinessException(ErrorCode.TICKET_SOLD_OUT,
            "티켓 재고가 부족합니다.");
    }
    
    // Gate 2: 연결된 세션 정원 검증
    List<Session> linkedSessions = getLinkedSessions(ticket);
    for (Session session : linkedSessions) {
        if (!session.hasCapacity(quantity)) {
            throw new BusinessException(ErrorCode.SESSION_CAPACITY_EXCEEDED,
                "세션 '" + session.getTitle() + "'의 정원이 초과되었습니다.");
        }
    }
    
    // Gate 3: 모집 상태 검증
    for (Session session : linkedSessions) {
        if (session.getRecruitmentStatus() != RecruitmentStatus.OPEN) {
            throw new BusinessException(ErrorCode.RECRUITMENT_CLOSED,
                "세션 '" + session.getTitle() + "'은 현재 모집중이 아닙니다.");
        }
    }
    
    // 통과 → 재고/정원 차감
    ticket.decreaseStock(quantity);
    for (Session session : linkedSessions) {
        session.increaseAttendees(quantity);
    }
    
    // 주문 생성
    return orderRepository.save(
        Order.create(userId, ticketId, quantity, ticket.getPrice())
    );
}
```

## 3. OrderService — 주문 취소 로직

> 원본 설계서 Section 3-2에서 발췌

```java
@Transactional
public void cancelOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    Ticket ticket = ticketRepository.findById(order.getTicketId());
    
    // 재고 복구
    ticket.increaseStock(order.getQuantity());
    
    // 세션 정원 복구
    List<Session> linkedSessions = getLinkedSessions(ticket);
    for (Session session : linkedSessions) {
        session.decreaseAttendees(order.getQuantity());
    }
    
    order.cancel();
}
```

## 4. 동시성 처리 — 비관적 잠금

> 원본 설계서 Section 7에서 발췌

인기 이벤트에서 동시에 여러 사용자가 구매를 시도할 때 **정원 초과 판매(Overselling)**를 방지해야 합니다.

### 비관적 잠금 쿼리 (OrderService에서 사용)

```java
// SessionRepository — event 도메인에서 제공 예정
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT s FROM EventSessionJpaEntity s WHERE s.id = :id")
Optional<EventSessionJpaEntity> findByIdForUpdate(@Param("id") Long id);

// TicketRepository — ticket 도메인에서 제공 예정
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT t FROM TicketJpaEntity t WHERE t.id = :id")
Optional<TicketJpaEntity> findByIdForUpdate(@Param("id") Long id);
```

### 트랜잭션 흐름

```
BEGIN TRANSACTION
  │
  ├─ SELECT ticket FOR UPDATE (비관적 잠금)
  ├─ SELECT sessions FOR UPDATE (비관적 잠금)
  │
  ├─ 재고 검증 → 실패 시 ROLLBACK + "매진" 응답
  ├─ 정원 검증 → 실패 시 ROLLBACK + "정원 초과" 응답
  │
  ├─ ticket.soldCount += quantity
  ├─ session.currentAttendees += quantity (각 세션)
  ├─ INSERT order
  │
COMMIT
```

## 8. 참고: 시나리오 상세 (정원 연동 설계서 발췌)

### 시나리오 1: 정상 구매

```
사전 상태:
  티켓 A (전체 세션) - 재고 100, 판매 0
  세션 1 - 정원 80, 참석 0
  세션 2 - 정원 60, 참석 0

사용자가 티켓 A를 2매 구매:
  → 티켓 A: 판매 2, 잔여 98
  → 세션 1: 참석 2, 잔여 78
  → 세션 2: 참석 2, 잔여 58
```

### 시나리오 2: 세션 정원으로 인한 자동 마감

```
사전 상태:
  티켓 A (전체: 세션1+세션2) - 재고 100, 판매 55
  티켓 B (개별: 세션1만) - 재고 50, 판매 20
  세션 1 - 정원 80, 참석 75 (잔여 5)   ← 거의 가득 참
  세션 2 - 정원 60, 참석 55

사용자가 티켓 B를 6매 구매 시도:
  → 세션 1 정원 검증 실패 (75 + 6 > 80)
  → "세션 'Day 1'의 정원이 부족합니다 (잔여 5석)"

사용자가 티켓 B를 5매 구매:
  → 세션 1: 참석 80, 잔여 0  ← 가득 참!
  → 티켓 A: isPurchasable = false (세션1 정원 초과)
  → 티켓 B: isPurchasable = false (세션1 정원 초과)
  → 두 티켓 모두 자동으로 "정원마감" 상태
```

### 시나리오 3: 주문 취소 후 복구

```
사전 상태:
  티켓 A - 판매 60, 잔여 40
  세션 1 - 참석 80, 잔여 0 ← 가득 참 (티켓 A, B 모두 마감)

사용자가 주문(티켓 A, 2매) 취소:
  → 티켓 A: 판매 58, 잔여 42
  → 세션 1: 참석 78, 잔여 2
  → 세션 2: 참석 58, 잔여 2
  → 티켓 A: isPurchasable = true ← 자동 복구!
  → 티켓 B: isPurchasable = true ← 자동 복구!
```
