# 🚀 VenueOn 프로젝트 개발 회고 및 핵심 설계 분석 리포트

작업하신 모든 PR 기록(약 65개 이상의 PR)과 Commit History를 분석하여, **주도적으로 설계하신 핵심 도메인 로직**과 **고도화 과정**, 그리고 **트러블 슈팅 내역**을 정리한 통합 리포트입니다. 포트폴리오나 이력서 작성 시 핵심 역량으로 바로 활용하실 수 있도록 체계화하였습니다.

## 1. 🏗️ 핵심 설계 및 고도화 로직

### 1-1. 복합 이벤트-세션-티켓 계층형 도메인 설계 (Hexagonal Architecture)
단순한 1:1 이벤트 구조가 아닌 **"이벤트(Event) > 여러 세션(Session) > 복수 티켓(Ticket)"** 이라는 복잡한 N:M 도메인 모델을 헥사고널 아키텍처(Port & Adapter)를 기반으로 설계했습니다.
- **도메인 분리 및 무결성 검증**: 이벤트를 단일, 복합, 온라인, 하이브리드 타입으로 세분화하고, 티켓 구매 시 특정 세션 혹은 전체 세션이 매핑되도록 설계했습니다.
- **UI 동기화 및 렌더링 최적화**: 복잡한 계층형 데이터를 프론트엔드에서 직관적으로 입력받을 수 있도록 Form 상태를 관리하였고, 사용자가 이벤트 생성 중 "티켓에 아무 세션도 연동하지 않는" 휴먼 에러를 방지하는 프론트엔드 사전 유효성 검사 로직(Validation)을 구축했습니다.

### 1-2. 코드(Enum) 데이터의 Numeric ID 기반 아키텍처 이전 (v7 Architecture)
하드코딩된 문자열 상태값(String ENUM) 시스템이 갖는 확장성의 한계를 파악하고, 백엔드의 모든 도메인 상태(이벤트, 결제, 모집상태 등)를 **Numeric ID(DB 식별자) 기반 참조 아키텍처**로 완벽하게 마이그레이션했습니다.
- **DTO 및 Mapper 최적화**: 프론트엔드 연동을 위해 단순히 숫자만 반환하지 않고, 백엔드 Mapper 과정에서 `{id: 숫자, label: 한글명}` 구조의 공통 `CodeDto`로 변환하여 응답하도록 설계했습니다. 이는 프론트엔드 단의 번거로운 변환 로직을 제거하고 렌더링 성능을 높이는 고도화된 접근입니다.

### 1-3. 프론트엔드 메타 레이아웃(Meta Layout) 추상화 및 캡슐화
여러 곳에 파편화되어 있던 UI 레이아웃 로직을 최상단 루트 레이아웃(Root Layout)과 컨테이너 CSS 컴포넌트로 규격화했습니다.
- **Smart Container 패턴**: `container-full`, `container-single`, `container-sidebar` 등 글로벌 CSS 구역을 분리하여 레이아웃을 모듈화했습니다.
- **AuthFormLayout 캡슐화**: 로그인, 회원가입, 호스트 회원가입, 비밀번호 찾기 등 인증 관련 UI가 각자 구현되던 것을 `AuthFormLayout`을 통해 일원화하여, 파편화된 디자인 코드의 응집도를 높이고 코드 중복을 획기적으로 줄였습니다.

### 1-4. HostProfile 도메인 분리 및 소셜(Oauth) 인증 연동
초기 사용자(User) 엔티티에 결합되어 있던 호스트 비즈니스 로직(사업자 등록 번호, 기관명 등)을 독립된 **HostProfile Entity로 1:1 분리(Decoupling)** 했습니다.
- 이로 인해 일반 사용자의 DB 부하를 줄였으며, Google 소셜 로그인과 커스텀 JWT 기반 인증 상태(Zustand)를 완벽하게 연동하여 사용자 및 호스트의 권한(Role)별 안전한 렌더링 라우팅 처리를 완성하였습니다.

## 2. 🛠️ 주요 트러블 슈팅 과정 (Trouble Shooting)

### 🚨 Trouble 1. SSR과 CSR 환경의 혼재로 인한 헤더 Login State 불일치 현상
* **문제 상황**: 사용자가 브라우저를 새로고침하거나 외부 링크를 통해 진입할 때, 토큰이 존재함에도 불구하고 헤더에서 로그인/로그아웃 버튼이 렌더링 전 깜빡이거나 잘못된 상태를 가리키는 하이드레이션(Hydration) 비동기화 문제가 발생했습니다.
* **해결 과정 (Trouble Shooting)**: 
  단순히 로컬 스토리지를 읽어오는 방식이 아닌, Zustand를 통한 비동기 상태 관리 (`useAuthStore`) 구조를 도입했습니다. `<Header />` 컴포넌트를 클라이언트 컴포넌트(`'use client'`)로 명시하고, 마운트 직후 `checkSession()`을 강제 호출하여 무효화된 기존의 꼬인 세션을 초기화함으로써 인증 상태 왜곡 현상을 영구적으로 해결했습니다.

### 🚨 Trouble 2. App Router 규격 오해 및 Spring Boot 파일 업로드 Size Limit 오류
* **문제 상황**: 이벤트 썸네일을 등록하는 과정에서 1000KB(1MB) 이상의 이미지가 지속적으로 `413 Payload Too Large` 에러와 함께 잘리는 현상이 발생.
* **해결 과정 (Trouble Shooting)**:
  1. 원인을 프론트엔드 Proxy인 `api/route.ts`로 추정하여 `export const config = { bodyParser: { sizeLimit: '10mb' } }` 구문을 추가했으나, Next.js **App Router에서는 이 설정이 지원되지 않음**을 파악하고 즉각 롤백.
  2. 에러의 기원지가 내부 Tomcat 인프라임을 확신하고 분석한 결과, Spring Boot의 기본 `multipart.max-file-size` 설정값이 1MB로 제한되어 있었음을 발견.
  3. `application.properties`의 `max-file-size`와 `max-request-size`를 모두 10MB로 명시적 확장하여 프록시 레이어 없이 인프라 계층에서 문제를 정확히 해결했습니다.

### 🚨 Trouble 3. 아키텍처 마이그레이션 중 스프링 DI(의존성 주입) 및 DataInit 충돌 오류
* **문제 상황**: 백엔드 시스템을 ENUM에서 Numeric ID로 v7 업그레이드하여 배포하던 도중 CI/CD 헬스체크가 실패하며 애플리케이션 시작이 실패하는 심각한 장애 발생.
* **해결 과정 (Trouble Shooting)**:
  1. 에러 로그 분석을 통해 `CommentAdminUseCase` Bean 생성 실패 및 `DataInitializer` 외래키 참조 에러를 포착.
  2. 아키텍처 마이그레이션 과정에서 누락되었던 **Interface 구현체(`CommentCommandService`)의 상속 구조를 재연결**하고, 분리된 Entity 스키마에 맞추어 `DataInitializer`의 더미 데이터 삽입 순서(의존성 순서) 및 코드 체계를 재정비하여 운영 서버 무중단 배포를 성공적으로 복구했습니다.

### 🚨 Trouble 4. Fixed(고정형) 헤더가 촉발한 글로벌 레이아웃 계층(Z-Index & Flow) 충돌 문제
* **문제 상황**: 메인 페이지의 심미적인 디자인을 위해 Header의 끝부분 모서리를 둥글게 깎으면서, 헤더를 `position: fixed`로 띄웠습니다. 이로 인해 메인 외의 모든 세부 페이지(로그인, 이벤트 등) 레이아웃이 헤더 너비만큼 위로 말려 올라가 심각하게 깨지는 부작용이 생겼습니다.
* **해결 과정 (Trouble Shooting)**:
  1. 모든 개별 페이지 컴포넌트에 Top 패딩을 하드코딩하려는 시도는 코드 응집도를 해칠 것이라 판단.
  2. Next.js의 `usePathname()` 훅을 활용해 헤더 내부에서 페이지를 판별하는 동적 CSS 렌더링 기법 도입.
  3. **메인 페이지(`/`)**에서는 `fixed`와 `HeroSection 상단 패딩`을 조합하여 투명도 오버랩 효과를 극대화하고, **나머지 모든 뷰포트**에서는 CSS `position: sticky` 동작으로 전환시켜 전역적인 레이아웃 이탈 문제를 근본적으로 차단하는 유연성을 확보했습니다.

## 💡 요약: 본인의 개발 성향 및 강점 
현재 코딩 스타일 및 해결 방식을 짚어보면, **에러가 발생했을 때 표면적인 UI/코드를 덧붙여 해결하는 것을 지양하고 원인 발생지(Root Cause)가 인프라/설계단인지 프론트/백엔드인지 명확히 파악하려는 '엔지니어링 마인드'**가 돋보입니다. 또한 CSS의 레이아웃 흐름 파악, Next.js의 SSR 생태계, Spring Boot의 헥사고널 아키텍처까지 훌륭한 **풀스택 이해도**를 보여주고 계십니다.
