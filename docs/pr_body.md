## 📌 변경 사항 내역 (진행한 내용)
1. **백엔드 업로드 폴더 계층 구조화 (스케일 업 대응)**
   - 평면적이었던 단일 파일 보관 방식(`backend/upload/...`)에서 벗어나 동적 연/월 폴더를 자동 생성하도록 `FileUploadController` 개선 (예: `event-thumbnail/2026/04/[UUID]`)
   - `DataInitializer` 시드 데이터 썸네일 경로 최신 구조 마이그레이션
2. **프론트엔드 환경 독립적 부분 상대 URL 적용**
   - 모든 프론트 페이지 파일 내 하드코딩된 API 환경변수(`${BACKEND_URL}`) 종속성 완전 제거
   - 브라우저가 현재 도메인을 기준으로 요청하도록 오직 상대 경로(`/upload/...`)만 던지게 변경
3. **로컬 및 배포 환경 최적화 서빙 파이프라인 분리 구축**
   - **배포 서버:** 운영 시 Spring의 스레드 점유를 차단하기 위해 호스트 서버에서 바로 이미지를 빼주는 고속 Nginx 정적 서빙 설정(`docs/nginx-venueon.conf`) 작성 완료.
   - **로컬 개발:** 개발 단계에서는 Nginx가 없기 때문에 프론트엔드 `next.config.ts` 의 `rewrites` 기능으로 우회 프록시를 뚫어 백엔드가 문제없이 제공하게끔 세팅.

## 🤔 진행하지 않은 내용 및 그 이유 (BFF Sharp 압축 전략 보류 사유)
원래 최초 기술 전략 문서에서는 **당장 Next.js BFF 환경에서 Sharp 라이브러리를 통해 WebP 압축을 실행하고 저장하는 방식**을 기획했습니다. 하지만 이를 현재 적용하지 않고 백엔드로 돌아간 이유는 다음과 같습니다:

1. **로컬 환경 개발 폴더 동기화 문제 (치명적):**
   - 배포 환경(Docker)과는 달리, 로컬 개발 터미널 환경에선 프론트엔드 프로세스와 백엔드 프로세스 간 물리적 볼륨(파일) 공유가 매우 까다롭고 불편함. 설정(`application.properties`)이 지저분해지고 매일 개발 꼬임 현상 발생 우려.
2. **데이터 결합도 및 개발 생산성 증대:**
   - 현재 구현 속도를 끌어올려야 하는 MVP 단계 특성 상, 데이터베이스에 정보를 씌우면서 이미지도 같은 백엔드에서 관할하면 트랜잭션 오류 시 삭제 처리 (롤백) 등이 직관적임.
3. **향후 고도화 여지 남김:**
   - 용량 절약을 위한 고사양 이미지 처리 로직은 추후 AWS S3 분리 등 인프라 아키텍처 스케일링이 실제 필요한 "퍼포먼스 튜닝 단계(Stage 2~3)"에서 일괄 전환(Refactoring)하는 것이 올바르다고 판단하여 전략 문서를 재현행화(`docs/이미지_업로드_관리_전략.md`) 하였음.
