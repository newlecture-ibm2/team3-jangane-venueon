# 🚀 VenueOn 프로젝트: 마스터 기술/UX 분석 리포트 (통합본)

> **작성일:** 2026-04-16  
> 본 문서는 VenueOn 프로젝트 기획의 배경과 핵심 기능부터, 내부 구조(백엔드/프론트엔드), 발생한 트러블 슈팅, 사용자 관점의 UX/UI 발표 자료까지 **모든 설계서와 분석 결과물(아키텍처, 포트폴리오 분석, 발표 대본)을 집대성하여 중복을 피하고 완벽하게 취합한 최종 통합 리포트**입니다.

## 1. 🧰 아키텍처 및 코어 엔지니어링 (Architecture & Engineering)

### 1-1. 모듈러 모놀리스 & 헥사고널 아키텍처 (Ports and Adapters)
프로젝트는 13개의 철저히 독립된 도메인 모듈(Event, Session, Ticket, Order, User, Community 등)로 분리되어 있습니다.
* **의존성 역전 (DI)**: Actor 계층(Host, Admin, Manager)은 데이터베이스에 직접 접근하지 않고, 코어 도메인의 Port(Interface)를 주입받아 모듈 간 강결합을 막습니다. 
* **Pub/Sub 이벤트 브로드캐스팅**: 알림(Notification) 로직의 경우 특정 모듈에 종속되지 않고 **이벤트 수신** 방식으로 동작하여 핵심 트랜잭션의 성능 부하를 방어합니다.

### 1-2. 코드 테이블 정규화 및 Numeric ID 시스템 구조화
단순 VARCHAR 형태의 하드코딩 Enum에 의존하던 코드 한계를 극복하기 위해, 사이트 내 모든 상태/유형을 **Numeric ID 기반 룩업 테이블(총 27개)로 전면 분리 정규화**했습니다.
* **CodeDto 캐싱 매퍼 도입**: 백엔드는 응답 시 단순 영문자("PUBLISHED")가 아닌 `{id: 2, label: "발행됨"}` 형태의 구조화된 코드 객체를 반환합니다. 이는 프론트엔드의 번역 리소스 관리를 근절시키고 추후 다국어 확장의 견고한 밑바탕이 되었습니다.

### 1-3. Soft Delete → Hard Delete 영속성 트러블 슈팅 (고유성 제약 대응)
* **문제**: 회원 탈퇴 시 안전을 위해 Soft Delete(`is_deleted=true`) 방식을 채택했으나, 탈퇴한 유저가 동일한 이메일로 다시 가입을 시도할 때 DB의 Unique Constraints(고유성 제약조건)가 터지는 버그가 발생했습니다.
* **해결**: 글로벌 개인정보 보호법(GDPR) 측면의 완벽한 파기를 설계 원칙으로 삼아 **Hard Delete(물리적 삭제)** 로 비즈니스 로직을 변경했습니다. 이에 따라 데이터 정합성을 해치지 않도록 외래키(Foreign Key) 계단식 삭제(Cascade) 처리까지 무결점하게 업데이트하고 도메인 모듈에 적용했습니다.

### 1-4. 의존성 충돌 및 DataInitializer 마이그레이션 트러블 슈팅
* 아키텍처를 ID 기반으로 이전하는 과정에서, `CommentAdminUseCase` Bean 주입 실패 및 Spring DataInit 과정에서의 참조(Reference) 충돌로 인해 서버 구동이 실패하는 헬스체크 데드락이 발생했습니다. 
* 이를 해결하기 위해 엔티티 연관관계를 재정립하고 Interface 생태계를 재연결하여 무중단 배포의 기초를 복구했습니다.

## 2. 🧠 고도화된 비즈니스 로직 (Advanced Business Logic)

### 2-1. "티켓 중심 판매" 복합 계층 분리 모델 (Event - Session - Ticket)
단순 세미나뿐 아니라 복합 컨퍼런스까지 담아낼 수 있는 패키징 시스템의 고민에서 출발했습니다.
* **Session**: "언제, 어디서, 최대 몇명이" 들을 수 있는지 시공간의 물리적 제약을 담당.
* **Ticket**: "어떻게 판매할 것인가"를 정의. 올패스 패키지부터 특정 일정만 참여하는 커스텀 패키지까지 호스트가 판매 단위를 유연하게 설계할 수 있게 분리했습니다.

### 2-2. 티켓-세션 간 "이중 게이트(Dual Gate)" 차감 시스템 & 비관적 잠금
인기 커리큘럼에서 발생할 수 있는 중복 판매(Overselling)를 차단하는 핵심 설계입니다.
* **비관적 잠금(Pessimistic Locking)** 을 통해 결제 동시성 환경에서의 트랜잭션을 보호합니다.
* 특정 티켓의 판매 가능 여부는 [그 티켓 자체 재고 + 묶여 있는 세션들의 잔여 정원]을 실시간으로 추적하는 **이중 게이트(Dual Gate) 검증**을 통해, 세션 하나라도 차면 엮여있는 모든 패키지 티켓을 자동으로 수강 마감 처리합니다.

### 2-3. 라이프사이클 "이중 모니터링" 체계 (Computed vs Forced / Real-time 반영)
이벤트의 진행/종료/마감을 판별하기 위해 수동 업데이트나 별도의 배치를 돌리지 않습니다.
* **Computed 자동 필터링**: 시스템 시간(Real-time)과 이벤트 일정 데이터를 상시 비교하여 **동적(자동)으로** 데이터를 산출합니다. 사용자가 마이페이지(MyPage)에 진입하면 자동으로 지난 이벤트들은 '진행 전/진행 중/종료' 탭으로 깔끔하게 자동 필터링 분류됩니다.
* **Forced (오버라이딩)**: 다만, 호스트의 돌발 상황으로 강제 마감할 시 `forced_session_status_id` 값을 업데이트하여 자동 연산을 덮어쓰고 지정 상태를 즉각 출력하는 안정장치(Fallback) 로직을 결합했습니다.

### 2-4. 다중 도메인 융합 권한(CommunityPermissionService) 시스템 구현
이벤트가 끝난 뒤에도 이어지는 네트워킹 생태계를 위해 단순 로그인 조회가 아닌 고도화된 검증 모듈을 제작했습니다.
* 게시글 접근/작성 시 흩어진 코드 검증을 피하기 위해 **`CommunityPermissionService`** 단일 서비스가 이를 전담합니다. 
* 결제 확인(`PAID`/`REGISTERED` 등 **Order 도메인** 체크) 또는 특정 클래스 수료 뱃지 보유 여부(**Badge 도메인** 체크) 등 이질적인 도메인의 상태를 실시간 통합 확인합니다. 미래에 '정기구독자 전용' 등 어떠한 비즈니스가 추가되어도 쉽게 Strategy 방식으로 끼워 넣을 수 있는 압도적 응집도와 유연성을 달성했습니다.

### 2-5. 상태 머신(State Machine) 기반 1:1 고객센터 루프 체계화
1:1 문의를 단순한 Q&A 게시판(CRUD)으로 두지 않았습니다.
* **PENDING(대기 중) → IN_PROGRESS(검토 중) → COMPLETED/REJECTED** 라는 상태 기반 라이프사이클로 고도화했습니다.
* 관리자가 글을 열어본 즉시 검토 중 상태로 변경되며, 이때부터는 유저가 자진 삭제/취소를 할 수 없게 하여 관리자와 유저 간의 처리 데드락(Deadlock)과 리소스 낭비를 없앴습니다. 반려(REJECTED) 시에도 사유 데이터를 확실히 매핑해 투명한 CS 루프를 완성했습니다.

## 3. 🎯 플랫폼 핵심 워크플로우 (Core Workflows)

### 3-1. 호스트 검토/승인 파이프라인 개입 모델 (환불)
* 단순 이커머스와 달리 일반적인 기능 구현을 탈피해 호스트(Host)와의 C2C 권한이 반영된 플랫폼 비즈니스 룰을 구성했습니다. 환불 신청 시 내역이 주최자 패널로 전송되고, **호스트(또는 어드민)이 최종 승인/반려 해야만 로직이 완료**되는 안전한 검토 구조를 삽입했습니다. 

### 3-2. "성장형 소셜 커뮤니티" 뱃지 리텐션 모델
* 일회성 지식 전달 기능(해당 강의 수강)에서 벗어나, 조건 완료 시 시스템이 **증명 홀로그램 뱃지**를 발급합니다.
* 뱃지 보유자만 진입할 수 있는 프라이빗 커뮤니티(자유로운 잡담~전문 지식 공유) 기능을 구축해 플랫폼 잔류도를 높였으며, 향후 링크드인(LinkedIn)처럼 유저들의 뱃지 수집과 성장을 타인에게 보여줄 수 있도록 **공개 여부(Visibility) 필드까지 선제적으로 설계**된 확장성 지향형 모델입니다.

### 3-3. 완벽히 분리된 역할 간 인가 대시보드 및 권한 위임 체계
* 최고 관리자(`/admin`), 주최자(`/host`), 위임 운영자(`/manager`) 의 워크스페이스 패널이 인트라넷처럼 완전히 격리됩니다. 커뮤니티 위임 운영자는 사이트 전체 권한 없이 특정 게시판에 대해서만 블라인드나 핀 기능을 수행할 수 있는 권한 위임(Delegation) 설계가 돋보입니다.

## 4. 🛠️ 성능 최적화 및 편의성 극대화 기능 (Backend/Frontend UX)

### 4-1. 비동기 설계 및 애플리케이션 포화(Threadpool) 방어
* 결제 직후 이어지는 메일링 발송(Toss 통신 및 SMTP 서버)이 동기 방식으로 처리되면서 화면 지연이 15초가량 멈추는(Blocking) 증상을 캐치했습니다. 화면에 즉시 피드백을 주기 위해 단순 알림/발송 등은 비동기 큐(`@Async`)를 거치도록 하되, 무분별한 큐 대기로 Tomcat 기본 스레드(200개)가 포화되어 다운되는 것을 막고자 시스템 분배 최적화를 고려한 엔지니어링 마인드가 적용되었습니다.

### 4-2. Google Calendar 'Slug' 즉각 연동 기능
* 단순 이메일 영수증 전송에서 끝나지 않고, 결제 확인 메일에 **구글 캘린더 자동 저장 슬러그 링크**를 포함했습니다. 사용자가 강의를 수강하자마자 버튼 1개만 눌러 자신의 캘린더에 동기화할 수 있도록 하여 오프라인 출석율과 UX를 대폭 상향시켰습니다.

### 4-3. 마이페이지 중앙 집중화(Aggregation)와 URL 파라미터 연동
* 찜, 관심사, 구매 이력 등이 각기 조각나지 않도록 메인 화면 등 전역에서 클릭된 하트(Wishlist)나 북마크 데이터가 **단일 마이페이지에 통합 아카이브(Aggregation)** 되는 API 구조를 짰습니다.
* 마이페이지 첫 화면의 요약 박스를 클릭하면, URL 파라미터를 읽어와서 동적으로 하위 탭이 즉시 열리게끔 하는 즉각 반응형 UI의 디테일을 추가했습니다.

### 4-4. 이메일 인증 절차 이탈 플로우 방지
* "이메일 인증을 끝내야만 가입이 완료"되는 무거운 스텝에서 기인하는 사용자 이탈(Drop-off)을 차단하고자, **가입 즉시 "비활성" 상태로 사용자 계정을 우선 생성**해두고, 추후 메일만 연결하면 바로 로그인 폼으로 스무스하게 직행하는 분리된 온보딩 설계를 채택했습니다.

### 4-5. 장바구니 중복 패키지 구매 방어 (Preventive Alert)
* 이벤트와 세션이 복잡한 패키징으로 판매되는 만큼 사용자가 실수로 동일 강의를 장바구니에 담는 '휴먼 에러' 확률이 존재함을 파악, 포함된 같은 세션이 장바구니에 진입할 경우 안내 메시지가 터지도록 하는 프레임워크 락을 걸어뒀습니다.

## 5. 🎨 렌더링 최적화 및 세분화된 UX 시각 제어 (Frontend UI)

### 5-1. 동적 레이아웃 계층 제어 (SSR Hydration / Scroll Overlay)
* **스마트 동적 헤더 제어 (Sticky vs Fixed)**: 메인 페이지에서는 히어로 배너와 겹쳐서 아름다운 풀사이즈 광원 렌더링을 낼 수 있도록 헤더를 `Fixed`로 주고, 타 상세 페이지나 입력 폼 등에서는 글자가 헤더 뒤로 숨지 않도록 Next.js 훅(`usePathname`) 판단 하에 **`Sticky` 모드로 브라우저 환경에서 자동 토글** 되도록 보정.
* **하이드레이션 깜빡임 박멸**: 로그인 정보를 서버/클라이언트 단 사이에서 일일이 읽어 렌더링 엇박자가 나던 헤더의 에러를 무력화. 헤더를 클라이언트 단위로 두고 진입 즉시 `checkSession()` 훅을 Store(Zustand)와 즉시 동기화해 상태 이탈을 원천 차단했습니다.

### 5-2. 메타 레이아웃(Meta Layout) 컨테이너 및 랩핑 추상화
* 개별 UI에서 일어나는 파편화를 잡기위해 글로벌하게 `container-full`, `container-sidebar` 등으로 구역을 지정. 특히 다양한 종류의 로그인/주최자가입/비번수정 등 인증 관련 코드를 반복하지 않고 하나의 **`<AuthFormLayout>`** 이란 부모로 감싸 UI 통일성과 관리 편의성의 응집도를 최대로 끌어올렸습니다.

### 5-3. 클라이언트 맞춤형 조건부 렌더링 (소셜 로그인 분기 등)
* 구글 OAuth 소셜 계정으로 회원가입한 사람에게 내부적인 '비밀번호 변경' 창이 나타나 시스템 혼동을 야기하는 것을 막기 위해, API 상의 인증 주체(`provider === 'LOCAL'`) 여부를 지속적으로 파악하여 **네이티브 가입자에게만 암호 체인징 버튼을 조건부 렌더링(Conditional Rendering)** 하는 UI 은닉 기술을 탑재했습니다.

### 5-4. 실시간 배지 UI 및 Octagonal 입체 뱃지 컴포넌트 
* 수동으로 판매일이나 상태를 적어주는 대신, 판매 기한에 따라 "D-3일 (마감임박 주황색 라벨)", "잔여 0석 (정원마감 회색 라벨)" 등이 즉각적으로 UI 엠블럼으로 계산되어 나타나 **직관적 가시성과 Urgency(희소성 재고) 마케팅 요소**를 극대화.
* 단순 박스형 CSS를 피하고 8각형 클립패스(Octagonal Clip-path) 기반의 파스텔톤 홀로그램 광원 효과를 넣은 리텐션 뱃지로 시각적 퀄리티 상승 입증.

### 5-5. 은닉형 로그아웃 및 논블로킹(Non-Blocking) 모달 제어 시스템
* 페이지 전환의 연속성을 끊는 투박한 `window.alert()`을 대체하여, 부드러운 전역 **Toast 알림과 Promise 기반 Confirm 비동기 모달**을 전면에 적용해 흐름이 깨지지 않게 보장했습니다.
* 로그아웃 버튼을 유저가 실수로 조작하기 쉬운 헤더 노출부 대신 프로필의 바닥(Depth 하단)으로 배치하여, 최근 대형 앱들(예: 인스타그램, 유튜브)처럼 세션 잔류시간을 아주 자연스럽게 늘려주는 UX의 '시각 심리학적 배치(Dark/Nudge Design)'까지 고려했습니다.
