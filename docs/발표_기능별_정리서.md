## 0. 🎬 프로젝트 도입 — 문제 제기

### 왜 VenueOn을 만들게 되었는가? [제안자: 정지은]

- **제안 내용**: 현재에도 많은 이벤트/교육 플랫폼들이 존재하지만, 근본적인 문제는 **기능들의 파편화와 경험의 단절**. 단순히 기술적으로 기능이 없는 게 아니라, **주최자/사용자/관리자가 서로 다른 언어로 대화하고 있는 것**이 진짜 문제점. [제안자: 정지은]
- **제안 내용**: 이러한 문제의식을 바탕으로 **주최자/사용자/관리자를 하나의 플랫폼에서 이어주고**, 이벤트가 끝난 후에도 서로의 경험을 이어가고 공유할 수 있는 플랫폼을 만들기 위해 각 팀원들과 분담하여 프로젝트를 진행. [제안자: 정지은]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 정지은 님의 문제 정의는 단순 기능 부재가 아닌 **"역할 간 소통 프로토콜의 불일치"**라는 본질적 이슈를 짚음
  - 이 관점이 VenueOn의 Admin/Host/User **3-Actor 아키텍처**로 직결되며, 각 역할별 독립된 워크스페이스(`/admin`, `/host`, `/my`)가 이 문제의식의 기술적 해답

## 1. 🏗️ 아키텍처 및 코어 엔지니어링

### 1-1. 모듈러 모놀리스 & 헥사고날 아키텍처 (Ports and Adapters)
- **제안 내용**: 프로젝트는 13개의 철저히 독립된 도메인 모듈(Event, Session, Ticket, Order, User, Community 등)로 분리. Actor 계층(Host, Admin, Manager)은 데이터베이스에 직접 접근하지 않고 코어 도메인의 Port(Interface)를 주입받아 모듈 간 강결합을 방지. [제안자: 조영일]
- **제안 내용**: Pub/Sub 이벤트 브로드캐스팅 — 알림(Notification) 로직은 특정 모듈에 종속되지 않고 이벤트 수신 방식으로 동작하여 핵심 트랜잭션의 성능 부하를 방어. [제안자: 조영일]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 아키텍처 v7 문서 검토 결과, 27개 테이블 / 176개 API / ~46개 페이지로 구성된 대규모 프로젝트임을 확인
  - 도메인 간 **ID 기반 느슨한 결합** (직접 클래스 참조 없음) 원칙이 일관되게 적용됨
  - Actor 계층의 Core Port 주입 패턴을 통해 엔티티/리포지토리 중복을 완전 차단한 점이 아키텍처적으로 우수
  - 기술 스택: Spring Boot 3.x + Next.js 16 + Vanilla CSS Module + PostgreSQL 15 + Redis 7

### 1-2. 코드 테이블 정규화 및 Numeric ID 시스템 (v7 아키텍처 전환)
- **제안 내용**: 단순 VARCHAR 형태의 하드코딩 Enum에 의존하던 코드 한계를 극복하기 위해, 사이트 내 모든 상태/유형을 **Numeric ID 기반 룩업 테이블(총 27개)**로 전면 분리 정규화. CodeDto 캐싱 매퍼 도입으로 백엔드는 응답 시 `{id: 2, label: "발행됨"}` 형태의 구조화된 코드 객체를 반환. [제안자: 조영일]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - ERD v7 분석 결과, `event_statuses`, `event_types`, `recruitment_statuses`, `user_roles` 총 4개의 신규 코드 테이블 추가 확인
  - `DomainCode(Long id, String label)` record 패턴으로 JPA → Domain → DTO 변환 파이프라인이 깔끔하게 정리됨
  - `CodeConstants` 클래스를 통한 상수 관리로 매직 넘버 방지
  - 프론트엔드의 번역 리소스 관리를 근절시키고 추후 **다국어 확장**의 견고한 기반 확보

> [!IMPORTANT]
> **트러블 슈팅: 아키텍처 마이그레이션 중 스프링 DI 및 DataInit 충돌 오류** [제안자: 조영일]
> - **문제**: ENUM에서 Numeric ID로 v7 업그레이드하여 배포하던 도중 CI/CD 헬스체크가 실패하며 애플리케이션 시작 실패. `CommentAdminUseCase` Bean 주입 실패 및 `DataInitializer` 외래키 참조 에러 발생
> - **해결**: Interface 구현체(`CommentCommandService`)의 상속 구조를 재연결하고, 분리된 Entity 스키마에 맞추어 `DataInitializer`의 더미 데이터 삽입 순서(의존성 순서) 및 코드 체계를 재정비하여 **운영 서버 무중단 배포를 성공적으로 복구**

## 2. 🔐 인증 및 보안 (JWT / OAuth2 / 회원 관리)

### 2-1. BFF(Backend For Frontend) 패턴 기반 JWT 인증 구조
- **제안 내용**: 브라우저 → Next.js 서버(BFF) → Spring Boot 3단계 인증. JWT는 iron-session으로 암호화되어 httpOnly 쿠키로만 저장. **브라우저에서 JWT에 접근 불가능** — XSS 공격에 대한 완벽한 방어. [제안자: 조영일]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - jwt-auth-flow.md 분석 결과, BFF 프록시(`app/api/[...path]/route.ts`)가 모든 API 요청에 JWT를 자동 주입하는 Catch-all 패턴 확인
  - 보안 체크포인트: `httpOnly: true`, `sameSite: 'lax'`(CSRF 방어), JWT 만료 시 자동 세션 삭제
  - Spring Security의 `JwtAuthenticationFilter` → `JwtTokenProvider.validateToken()` → `SecurityContext` 등록 → 역할 기반 인가(`/admin/**` → `ROLE_ADMIN`) 흐름이 완결적
  - Middleware(`frontend/middleware.ts`)가 페이지 접근을 1차 차단하고, 실제 JWT 유효성은 Spring Boot가 API 요청마다 2차 검증하는 **이중 방어 구조**

---

### 2-2. 이메일 인증 절차 이탈 플로우 방지
- **제안 내용**: 회원가입할 때 이메일 인증 때문에 가입 도중 페이지를 벗어나는 플로우가 불편하다고 생각. **가입 즉시 "비활성" 상태로 계정을 우선 생성**해두고, 이메일로 인증 링크를 보내 클릭 시 바로 로그인 페이지로 이동할 수 있게 플로우를 구성. [제안자: 조영일]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 사용자 이탈(Drop-off)을 차단하는 분리된 온보딩 설계로, 무거운 인증 스텝으로 인한 가입 중단율을 줄이는 효과
  - `is_active` 필드를 활용한 비활성 회원 → 인증 완료 시 활성화 전환 패턴

---

### 2-3. 소셜 로그인과 회원정보(비밀번호) 탭 분리
- **제안 내용**: 구글 OAuth 소셜 로그인의 경우 OAuth Provider(Google)가 인증을 위임받기 때문에 내부적인 패스워드가 무의미. 유저의 인증 공급자(Provider) 타입을 API를 통해 판별하여, `provider === 'LOCAL'`일 때만 `/security` 탭을 조건부 렌더링(Conditional Rendering). [제안자: 박단희]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - HostProfile 도메인이 User에서 1:1 분리(Decoupling)되어 일반 사용자의 DB 부하를 줄이는 설계가 portfolio_analysis에서 확인됨
  - Google 소셜 로그인과 커스텀 JWT 기반 인증 상태(Zustand)를 완벽 연동한 사용자 및 호스트의 **권한(Role)별 안전한 렌더링 라우팅** 완성

### 2-4. 회원 탈퇴 처리 — Soft Delete → Hard Delete 전환

> [!WARNING]
> **트러블 슈팅: Soft Delete → Hard Delete 영속성 트러블 슈팅** [제안자: 박단희]
> - **문제**: 회원 탈퇴 시 안전을 위해 Soft Delete(`is_deleted=true`) 방식으로 구현했으나, 탈퇴한 유저가 동일한 이메일로 다시 가입을 시도할 때 DB의 **Unique Constraints(고유성 제약조건) 오류**가 발생
> - **해결**: 글로벌 개인정보 보호법(GDPR) 측면의 완벽한 파기를 설계 원칙으로 삼아 **Hard Delete(물리적 삭제)**로 비즈니스 로직을 변경. 외래키(Foreign Key) 계단식 삭제(Cascade) 처리까지 무결점하게 업데이트

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 프로젝트 기술 분석 리포트 확인 결과, 해당 트러블 슈팅은 단순 버그 수정이 아닌 **법적 준수(GDPR) + DB 무결성 + 사용자 경험**을 동시에 충족하는 3중 해결
  - Cascade 삭제 시 관련 도메인(주문, 뱃지, 커뮤니티 멤버십 등) 전체에 대한 연쇄 영향 분석이 필요했을 것

## 3. 📅 이벤트 · 세션 · 티켓 시스템 (핵심 도메인)

### 3-1. Event - Session - Ticket 복합 계층 분리 모델
- **제안 내용**: 세미나 같은 이벤트에서 강의장/날짜별로 다른 세미나 일정으로 세션이 나뉘면 어떻게 상세페이지에서 사용자에게 한눈에 보여줄 수 있을지, 패키지로 이벤트를 구성해서 한번에 여러 이벤트를 할인해서 제공하고 싶을 때는 어떻게 할지 고민. [제안자: 조영일]
- **제안 내용**: 세션으로 내 이벤트를 자유롭게 구성하고 티켓으로 세션을 묶음 단위로 제안하여 호스트가 자유롭게 패키징을 커스텀할 수 있게 만듦. [제안자: 조영일]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 세션_판매_전략_설계서에서 3개 방안(A: TicketType / B: Bundle / C: 하이브리드)을 비교 분석하여 **방안 C(TicketOption + Smart Default)**를 최종 권장
  - 핵심 설계 원칙:
    - **Session** = "언제, 어디서, 최대 몇명" (시공간의 물리적 제약)
    - **Ticket** = "얼마에, 어떤 세션에 입장할 수 있는가" (판매 상품)
  - `SalesMode` enum으로 ALL_IN_ONE(통째 판매), INDIVIDUAL(개별 판매), BUNDLE(묶음 판매), SELECTIVE(선택적 판매) 4가지 전략 지원
  - `TicketOptionAutoGenerator`를 통한 Smart Default 자동 생성으로 호스트의 인지 부하 최소화

---

### 3-2. 티켓-세션 간 "이중 게이트(Dual Gate)" 차감 시스템 & 비관적 잠금
- **제안 내용**: 인기 커리큘럼에서 발생할 수 있는 중복 판매(Overselling)를 차단하는 핵심 설계. 비관적 잠금(Pessimistic Locking)을 통해 결제 동시성 환경에서의 트랜잭션을 보호. 특정 티켓의 판매 가능 여부는 [그 티켓 자체 재고 + 묶여 있는 세션들의 잔여 정원]을 실시간으로 추적하는 이중 게이트(Dual Gate) 검증. [제안자: 조영일]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 세션_판매_전략_설계서의 정원 관리 로직 분석 결과, 묶음 티켓과 개별 티켓이 동시에 같은 세션의 정원에 영향을 미치는 핵심 이슈를 `@Transactional` + 비관적 락으로 해결
  - 향후 트래픽 증가 시 Redis 기반 재고 관리로 전환 가능한 확장 경로가 설계됨
  - 세션 하나라도 차면 엮여있는 모든 패키지 티켓을 자동으로 **수강 마감 처리**하는 연쇄 검증 로직 존재

---

### 3-3. 이벤트 상태 관리 — Computed + Forced 이중 모니터링 체계
- **제안 내용**: 이벤트의 진행/종료/마감을 판별하기 위해 수동 업데이트나 별도의 배치를 돌리지 않음. 시스템 시간(Real-time)과 이벤트 일정 데이터를 상시 비교하여 동적(자동)으로 데이터를 산출. [제안자: 조영일]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 이벤트_상태관리_설계서 전체 분석 완료. 핵심 원칙: **"세션이 유일한 Source of Truth"**
    - 이벤트 시작일 = `min(세션들의 startTime)`, 종료일 = `max(세션들의 endTime)`
    - 이벤트 진행 상태 = 세션들의 진행 상태를 **OR 종합**
  - **EventStatus** 5단계: DRAFT → PUBLISHED → ONGOING → ENDED / CANCELLED
  - **RecruitmentStatus** 3단계: PENDING → OPEN → CLOSED
  - **v7 확장**: `forced_session_status_id` / `forced_recruitment_status_id`를 통해 호스트가 Computed 결과를 **수동 오버라이딩** 가능 (돌발 상황 대응)
  - 모든 날짜 필드는 nullable — 미설정 시 자동 전이 없이 호스트 수동 관리 모드로 전환
  - DB 스키마 변경: `events` 테이블에서 `start_date`/`end_date` 컬럼 제거, `event_sessions`에 모집 관련 3개 컬럼 추가

## 4. 💳 결제 · 장바구니 · 메일링

### 4-1. 토스 페이먼츠 연동 및 결제 완료 메일 발송
- **제안 내용**: 결제 직후 이어지는 메일링 발송(Toss 통신 및 SMTP 서버)이 동기 방식으로 처리되면서 화면 지연이 **10~15초**가량 멈추는(Blocking) 증상을 캐치. [제안자: 이규원]

> [!WARNING]
> **트러블 슈팅: 결제 완료 후 페이지 로딩 지연 (동기→비동기 전환)** [제안자: 이규원]
> - **문제**: 백엔드의 결제완료 메일 발송 메서드가 동기 방식으로 실행되어, 한 스레드가 토스 서버, SMTP 서버에 접속하고 대기하며 한번에 모든 일을 처리 → 결제 완료 페이지 로딩이 10~15초
> - **분석**: 비동기 방식으로 바꾸면 화면 지연은 해결되지만, 결제 요청이 동시에 1000/10000개가 될 경우 무한정 많은 queue가 생겨 **메모리 저장공간 부족으로 서버 다운** 위험. 스프링부트의 톰캣은 스레드를 기본 200개로 사용하고 있으며, 비동기 방식을 무분별하게 사용하면 톰캣이 가진 서버 다운 방어막을 제대로 쓰지 못하는 상황
> - **해결 방향**: 무작정 비동기로 바꾸기보다 비동기로 바꾼 후 **스트레스 테스트를 진행하며** 시스템 분배 최적화를 고려한 엔지니어링 마인드를 적용

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 프로젝트_기술_분석_리포트에서도 이 이슈를 "비동기 설계 및 애플리케이션 포화(Threadpool) 방어" 항목으로 다룸
  - `@Async` 큐를 거치되, Tomcat 기본 스레드(200개) 포화 방지를 위한 시스템 분배 최적화가 핵심
  - **상호 보완 지점**: 이규원 씨의 현장 경험(문제 발견 → 분석 → 해결 전략 수립)과 기술 분석 리포트의 아키텍처 관점(Threadpool 방어 전략)이 정확히 일치

- **제안 내용**: 권한 검증 로직을 전담하는 `CommunityPermissionService`를 구축. 게시글을 쓸 때나 페이지에 진입할 때 등, 권한이 필요한 모든 시점에서 이 서비스를 호출하도록 구조를 단일화. 비즈니스 로직과 권한 검증 로직을 명확히 분리. [제안자: 장혜은]
- **제안 내용**: 향후 구독제나 레벨 시스템 같은 새로운 커뮤니티 타입이 추가되더라도, 기존 로직을 건드리지 않고 `CommunityType` 전략만 추가하면 되는 구조를 완성. [제안자: 장혜은]

### 6-6. 1:1 문의 — 상태 머신(State Machine) 기반 CS 루프

  - **PENDING(대기 중)** → 사용자 문의 작성
  - **IN_PROGRESS(검토 중)** → Admin이 패널에서 데이터를 조회하는 즉시 상태 전환
  - 검토 중 이전까지만 유저가 **자진 취소(Soft Cancel)** 가능 → 관리자의 데드락 방지
  - **승인(처리 완료)** 또는 **반려(REJECTED)** → 반려 시 Admin이 남긴 사유를 매핑해 CS 루프 완성

## 7. 🖥️ 관리자(Admin) · 호스트(Host) 대시보드

### 7-1. 호스트 대시보드 — 호스트 친화적 운영 환경 구축
- **제안 내용**: 호스트에게 최대한 친화적인 환경을 만드는 데 집중. 호스트가 접속하자마자 운영 현황을 한눈에 파악할 수 있도록 대시보드를 구성하고, 필요할 때 항목별 상세 정보를 바로 확인할 수 있도록 기능을 구현. [제안자: 이세기]
  - **대시보드 `/host`**: 접속 직후 운영 요약(내 이벤트, 임시저장, 최근 주문, 주문 요약)을 바로 조회. 주요 페이지(이벤트 목록, 수강생 관리, 주문 관리, 새 강의 개설, 프로필, 문의)로 즉시 이동
  - **이벤트 관리**: 목록(`/host/events`) — 전체/임시저장/상태별 조회. 상세(`/host/events/{id}`) — 상세 정보 + 수강생 탭
  - **주문/정산 관리**: 주문 목록(`/host/payments`) — 상태/이벤트 필터. 주문 상세(`/host/payments/{id}`) — 주문 상세 확인 후 이벤트 상세로 역이동 가능
  - **수강생 통합 관리 `/host/attendees`**: 이벤트별/이름 필터로 수강생 목록 조회
  - **프로필 `/host/profile`**: 조회/수정, 이미지 업로드
  - **전체 흐름**: 로그인 → `/host` 대시보드 → (이벤트/주문/수강생/프로필/문의) → 상세/수정/생성

### 7-2. 어드민 대시보드 — 플랫폼의 흐름을 제어하는 실시간 모니터링 시스템
- **제안 내용**: VenueOn에서 관리자는 **실시간 모니터링을 하는 제어자**. 플랫폼의 흐름을 제어하는 어드민 대시보드를 구축하고, 플랫폼의 현재 사용통계를 한눈에 볼 수 있게 차트화. 불량 사용자 제재부터 문의 처리까지 실시간으로 대응하여 **플랫폼의 신뢰도를 기술적으로 보장**. [제안자: 정지은]
- **제안 내용**: 회원 관리(정지/권한 변경/삭제), 카테고리 관리, 신고 관리(처리 단계 + 제재 상태 + 이의 제기 + 이력 추적), 커뮤니티 요청 관리, 요청 처리, 리뷰 관리, 환불 관리, 커뮤니티 제재 관리 등 종합 관리 기능. [제안자: 조영일]

#### ① 데이터 정합성 중심의 사용자 및 관리 시스템 [제안자: 정지은]
- **Soft Delete & Cascade**: 데이터 무결성을 위해 논리적 삭제를 도입. JPA Cascade 설정을 최적화하여 사용자가 탈퇴하더라도 **서비스 통계에 필요한 연관 데이터는 안전하게 보존**되도록 설계.
- **재가입 로직 최적화**: 탈퇴 후 재가입 시 발생하는 식별값 충돌 문제를 해결하기 위해 인증 플로우를 고도화하여 **사용자 생애주기 전반의 보안을 강화**.

#### ② 운영 효율을 위한 통합 이벤트 관리 — JPA Specification 활용 [제안자: 정지은]
- **복잡한 동적 쿼리 대응**: 모집 상태, 카테고리, 키워드 등 4가지 이상의 복잡한 필터 조건을 처리하기 위해 **JPA Specification**을 도입. 단순 if-else 기반 쿼리 생성의 한계를 극복하고, 조건문을 객체화하여 **재사용성과 타입 안정성**을 동시에 확보.
- **실수 방지 UX**: 삭제 대신 **'숨김' 기능을 우선 제공**하고, 중요 상태 변경 시 **Step-wise Modal**을 적용해 관리자의 조작 실수를 원천 차단.

> *"단순히 관리자 페이지를 하나 만든 것에 그치지 않고, **서비스가 지속 가능하기 위해 데이터가 어떻게 관리되어야 하는가?**에 대한 고민을 아키텍처와 로직에 녹여내려 노력했습니다."* — 정지은

> [!WARNING]
> **트러블 슈팅: 숨김/삭제 처리 시 UI-DB 상태 싱크 불일치** [제안자: 정지은]
> - **문제**: 관리자가 호스트/유저/이벤트를 숨김이나 삭제 처리하는 과정에서, 백엔드에서 데이터 무결성 로직을 만들고 프론트로 이어줄 때 상태 변경이 **즉시 반영되지 않거나 잘못된 정보가 남는 문제** 발생
> - **해결**: Soft Delete를 도입하고 **UI와 DB의 싱크를 맞추는 데 집중**. 상태 변경 후 프론트엔드에서 즉시 리페치(re-fetch)하여 데이터 정합성 확보

> [!WARNING]
> **트러블 슈팅: 사용자 제재(비활성화) → 활성화 복원 로직** [제안자: 정지은]
> - **문제**: 가입한 유저/호스트를 제재(비활성화)한 후 다시 활성화로 복원하는 과정에서 상태 전이 로직의 복잡도가 높았음
> - **해결**: 상태 전이를 명확한 플래그 기반으로 관리하고, 관련 연관 데이터(주문, 커뮤니티 멤버십 등)의 정합성까지 함께 검증하는 복원 파이프라인 구축

> [!NOTE]
> **트러블 슈팅: 헥사고날 아키텍처 룰셋 준수와 팀 협업의 병행** [제안자: 정지은]
> - **문제**: 헥사고날 아키텍처를 이용하여 최대한 룰셋을 지키면서 구현해야 했으나, AI가 생성하는 코드를 100% 이해하면서 진행하기에 벅찬 부분이 존재. 팀프로젝트이기 때문에 **다른 팀원의 도메인들과 꼬이지 않게** 작업할 필요가 있어 코드를 항상 확인해야 했음
> - **배운 점**: MVP와 추가 기능의 구분을 초기에 명확히 하는 것의 중요성을 체감. 구조를 맞추면서 진행하며 **코드 리뷰와 도메인 간 경계 확인**의 중요성을 실무적으로 학습
> - **향후 과제**: 경고 기능, 신고 기능, 알림 기능 등의 로직 구현은 **구조적인 부분과 함께 추가 학습이 필요**하다고 판단

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 어드민 페이지 10개(`/admin/*`)로 구성되어 회원 정지/권한, 카테고리, 신고, 환불, 커뮤니티 제재 관리까지 포괄

- **제안 내용**: 파일 업로드의 수신 · 저장 · 서빙 · 구조화 전략. [제안자: 조영일]

    - **업로드**: Spring Boot → 디렉토리 순회(`../`) 공격 방지 검증 → 연/월 분류 폴더 동적 생성 → Volume에 저장
    - **서빙(로컬)**: Spring `WebMvcConfig` + `next.config.ts` rewrite 연동
    - **서빙(운영)**: Host Nginx가 `/upload/**` 경로를 호스트 Volume에서 직접 읽어 서빙 — **Backend 스레드 점유 방지 및 고속 응답 (0.5ms)**

> [!IMPORTANT]
> **트러블 슈팅: App Router 규격 오해 및 Spring Boot 파일 업로드 Size Limit 오류** [제안자: 조영일]
> - **문제**: 이벤트 썸네일 등록 시 1MB 이상의 이미지가 `413 Payload Too Large` 에러와 함께 잘림
> - **시도 1**: 프론트엔드 Proxy인 `api/route.ts`에 `bodyParser.sizeLimit: '10mb'` 추가 → Next.js **App Router에서는 이 설정이 미지원**임을 파악하고 즉각 롤백
> - **시도 2**: 에러의 기원지가 내부 Tomcat 인프라임을 확신하고 분석 → Spring Boot의 기본 `multipart.max-file-size` 설정값이 1MB 제한이었음 발견
> - **해결**: `application.properties`의 `max-file-size`와 `max-request-size`를 모두 10MB로 명시적 확장 → 프록시 레이어 없이 **인프라 계층에서 정확히 해결**

## 9. 🎨 프론트엔드 UX/UI 최적화

### 9-1. 동적 헤더 제어 (Fixed vs Sticky)
- **제안 내용**: 메인 페이지에서 히어로 배너와 겹쳐서 아름다운 풀사이즈 광원 렌더링을 위해 헤더를 `Fixed`로 설정했으나, 타 상세 페이지에서 글자가 헤더 뒤로 숨는 문제 발생. [제안자: 조영일]

> [!WARNING]
> **트러블 슈팅: Fixed 헤더가 촉발한 글로벌 레이아웃 계층(Z-Index & Flow) 충돌** [제안자: 조영일]
> - **문제**: 메인 페이지의 심미적 디자인을 위해 헤더를 `position: fixed`로 띄우자, 메인 외 모든 세부 페이지의 레이아웃이 헤더 너비만큼 위로 말려 올라가 심각하게 깨짐
> - **시도**: 모든 개별 페이지에 Top 패딩을 하드코딩하려 했으나 코드 응집도를 해칠 것이라 판단
> - **해결**: Next.js `usePathname()` 훅으로 페이지를 판별하여 **메인 페이지(`/`)에서는 `fixed`**, **나머지 모든 뷰포트에서는 `sticky`**로 자동 토글하는 유연한 동적 CSS 렌더링

### 9-2. SSR 하이드레이션 깜빡임 박멸

## 10. 🚀 서비스 비전 및 확장성

### 10-1. 이벤트 기반 서비스의 확장 가능성
- **제안 내용**: 지금은 강의와 세미나 형식에 초점을 두어 개발했지만, 기본적으로는 이벤트를 기준으로 만든 서비스이기 때문에 확장 가능성이 높음. 원하는 주제로 약간의 고도화만으로도 쉽게 확장 가능. 확장하면 점점 더 커뮤니티가 커지면서 사용자를 많이 묶어둘 수 있음. [제안자: 조영일]

- **제안 내용**: 한 이벤트의 주제를 가지고도 진지한 지식공유, 재밌는 잡담공유를 하고싶은 사람들끼리 커뮤니티를 형성할 수 있음. [제안자: 조영일]

- **제안 내용**: 우리도 일방향적으로 강사님의 수업을 들었을 때보다 슬랙에서 서로 이야기하며 지식과 관심사를 공유했을 때 더 좋은 효과가 있었음. [제안자: 이규원]

- 💡 **분석가 인사이트**: [분석자: Young Il]
  - 이벤트_상태관리_설계서의 장기 로드맵: 자동 리마인더, 반복 이벤트, 시간대(Timezone) 지원, 얼리버드/단계별 가격 등이 설계됨
  - 세션_판매_전략_설계서의 Phase 3: 얼리버드 할인, 수량 한정 티켓, 할인 코드/프로모션, 대기열(Waitlist) 기능까지 확장 경로가 명시
  - 이미지 업로드의 Stage 3(AWS S3 + CloudFront)까지 인프라 확장 경로도 준비됨
