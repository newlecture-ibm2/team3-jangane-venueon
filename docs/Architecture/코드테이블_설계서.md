# 코드 테이블(Code Table) 설계서 — 하이브리드 전략

> **작성일:** 2026-04-13  
> **목적:** ERD v6에서 VARCHAR로 하드코딩된 상태/유형/역할 값들을 **코드 테이블** 또는 **Enum**으로 체계적으로 분리 관리  
> **전략:** 확장 가능성이 높은 필드 → 코드 테이블 / 시스템 로직 직결 필드 → Enum 유지

## 3. 신규 요구사항 — MANAGER 역할 추가

### 3.1 배경

커뮤니티 관리 전담 역할(`MANAGER`)을 신설합니다.  
기존 `ADMIN`(전체 시스템 관리자)과 분리하여, **커뮤니티 도메인에 한정된 관리 권한**을 부여합니다.

### 3.2 USER_ROLE 코드 변경

| code_key | label | 설명 |
|----------|-------|------|
| `ADMIN` | 시스템 관리자 | 전체 시스템 관리 (기존) |
| `MANAGER` | 커뮤니티 관리자 | **🆕 신규** — 커뮤니티 관련 관리 권한 |
| `HOST` | 호스트 | 이벤트 생성/운영 (기존) |
| `USER` | 일반 사용자 | 참가/구매 (기존) |

### 3.3 MANAGER 권한 범위

| 도메인 | 권한 | 설명 |
|--------|------|------|
| 커뮤니티 | CRUD + 승인/반려 | 커뮤니티 생성 승인, 운영 규칙 관리 |
| 게시글 | 숨김/삭제 | 부적절 게시글 관리 |
| 댓글 | 숨김/삭제 | 부적절 댓글 관리 |
| 커뮤니티 멤버 | 역할 변경/차단 | 멤버 관리 (SUB_ADMIN 임명, BLOCKED 처리) |
| 신고(커뮤니티) | 처리 | POST/COMMENT 대상 신고 처리 |
| 이벤트 | ❌ 접근 불가 | 이벤트 관리는 ADMIN/HOST 전용 |
| 사용자 관리 | ❌ 접근 불가 | 계정 정지 등은 ADMIN 전용 |

## 4. 코드 테이블 DDL

### 4.1 테이블 구조

```sql
-- 코드 그룹 마스터
CREATE TABLE code_groups (
    id          BIGSERIAL PRIMARY KEY,
    group_key   VARCHAR(50) NOT NULL UNIQUE,
    name        VARCHAR(100) NOT NULL,
    description VARCHAR(255),
    is_system   BOOLEAN NOT NULL DEFAULT true,
    created_at  TIMESTAMP NOT NULL DEFAULT now()
);

-- 개별 코드
CREATE TABLE codes (
    id          BIGSERIAL PRIMARY KEY,
    group_id    BIGINT NOT NULL REFERENCES code_groups(id) ON DELETE CASCADE,
    code_key    VARCHAR(50) NOT NULL,
    label       VARCHAR(100) NOT NULL,
    description VARCHAR(255),
    sort_order  INT NOT NULL DEFAULT 0,
    is_active   BOOLEAN NOT NULL DEFAULT true,
    extra       VARCHAR(500),
    created_at  TIMESTAMP NOT NULL DEFAULT now(),
    updated_at  TIMESTAMP,
    UNIQUE (group_id, code_key)
);

CREATE INDEX idx_codes_group_id ON codes(group_id);
CREATE INDEX idx_codes_active ON codes(group_id, is_active);
```

### 4.2 컬럼 설명

| 테이블 | 컬럼 | 설명 |
|--------|------|------|
| code_groups | `group_key` | 시스템 식별키 (예: `USER_ROLE`) |
| code_groups | `is_system` | true면 관리자도 그룹 자체 삭제 불가 |
| codes | `code_key` | 시스템 식별키 (예: `ADMIN`) — 비즈니스 로직에서 사용 |
| codes | `label` | 프론트 표시명 (예: "시스템 관리자") |
| codes | `extra` | 부가 JSON (색상, 아이콘, severity 등) |
| codes | `is_active` | false로 소프트 삭제 (기존 참조 보호) |

### 8.3 JPA Entity 전환 패턴

**Before:**
```java
@Enumerated(EnumType.STRING)
@Column(name = "status")
private EventStatus status;
```

**After:**
```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "status_code_id")
private CodeJpaEntity statusCode;

// 편의 메서드 — 비즈니스 로직에서 code_key로 비교
public String getStatusKey() {
    return statusCode != null ? statusCode.getCodeKey() : null;
}
```

### 8.6 코드 캐시 전략

```java
@Service

    }

    @CacheEvict(value = "codes", allEntries = true)
    public void evictAll() {}  // 관리자 코드 수정 시 호출
}
```

## 11. 데이터 마이그레이션 (운영 환경)

```sql
-- 1) 코드 테이블 생성 + Seed 데이터 삽입
-- 2) 기존 테이블에 FK 컬럼 추가 (nullable)
ALTER TABLE users ADD COLUMN role_code_id BIGINT REFERENCES codes(id);

-- 3) 기존 문자열 → 코드 ID 매핑
UPDATE users u
SET role_code_id = c.id
FROM codes c
JOIN code_groups cg ON c.group_id = cg.id
WHERE cg.group_key = 'USER_ROLE' AND c.code_key = u.role;

-- 4) NOT NULL 제약 추가
ALTER TABLE users ALTER COLUMN role_code_id SET NOT NULL;

-- 5) 기존 문자열 컬럼 삭제
ALTER TABLE users DROP COLUMN role;
```

> [!CAUTION]
> 운영 환경에서는 2~3단계를 먼저 배포하여 **양쪽 컬럼이 공존하는 상태**에서 검증한 뒤,  
> 4~5단계를 별도 배포로 적용합니다. 롤백 안전성을 확보하기 위함입니다.
