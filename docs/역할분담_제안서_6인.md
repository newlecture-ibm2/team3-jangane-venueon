## 📐 1. 분배 원칙 및 이유

### 원칙 1: 도메인 수직 분할 (Vertical Slicing)

```
❌ 수평 분할 (역할별)              ✅ 수직 분할 (도메인별)
┌──────────────────────┐        ┌──────┬──────┬──────┬──────┐
│  FE 담당 3명          │        │ 인증  │이벤트│커뮤  │게시판│
│  (모든 페이지)        │        │ BE+FE│BE+FE│BE+FE│BE+FE │
├──────────────────────┤        │  A   │  B  │  D  │  F   │
│  BE 담당 3명          │        │      │     │     │      │
│  (모든 API)           │        │      │     │     │      │
└──────────────────────┘        └──────┴──────┴──────┴──────┘
→ FE↔BE 인터페이스 합의 必        → 한 사람이 API+UI 모두 책임
→ 커뮤니케이션 비용 ↑              → 커뮤니케이션 비용 ↓
→ 같은 파일 동시 수정 빈번         → 파일 충돌 거의 없음
```

**이유:**
- 헥사고날 아키텍처 + 모듈러 모놀리스 구조에서는 도메인 모듈(`user/`, `event/`, `community/`)이 **패키지 단위로 완전히 분리**되어 있음
- 한 사람이 백엔드 API와 프론트엔드 UI를 모두 담당하면 **BE↔FE 인터페이스 합의 불필요** → 개발 속도 ↑
- Next.js의 App Router 구조도 **폴더 기반 라우팅**이라 도메인별로 폴더가 나뉘어 있어 충돌 없음

### 원칙 2: 공통 모듈 전담자 지정

**이유:**
- `common/` 패키지(ApiResponse, Security, JWT 필터 등)와 공통 UI(`Header`, `Footer`, `Button` 등)는 **모든 팀원이 의존**하는 핵심 기반
- 여러 명이 동시에 수정하면 **머지 충돌의 주범**이 됨
- **1명이 전담**하여 Sprint 0에서 빠르게 완성 → 나머지 팀원은 안정된 기반 위에서 개발

### 원칙 3: event/ · community/ 모듈 내부를 파일 단위로 분할

**이유:**
- `event/` 모듈은 이벤트 CRUD + 주문/결제를 포함 → **B·C 2명**이 파일 단위 분할
- `community/` 모듈은 커뮤니티 관리 + 게시글/댓글을 포함하여 **작업량이 16 API로 최대** → **D·F 2명**이 파일 단위 분할
- 같은 패키지 안이지만 `CommunityController.java`와 `PostController.java`처럼 **파일이 다르므로 Git 충돌 없음**

### 5인 → 6인 변경 이유

| 항목 | 5인 체제 | 6인 체제 |
|------|---------|---------|
| **D 담당** | community 모듈 전체 (16 API) | 커뮤니티 CRUD + 멤버 (8 API) |
| **F 담당** | — | 게시글 + 댓글 CRUD (8 API) |
| **D 부담** | API 16개 (최다) | API 8개 (균형) |
| **충돌 위험** | 없음 | 없음 (파일 완전 분리) |

## 🛡️ 5. 충돌 방지 규칙

### Git 브랜치 전략

```
main
└── develop
    ├── feature/auth          ← A 전용
    ├── feature/event         ← B 전용
    ├── feature/order-host    ← C 전용
    ├── feature/community     ← D 전용
    ├── feature/common-ui     ← E 전용
    └── feature/post-comment  ← F 전용
```

- 각자 **자신의 feature 브랜치에서만 작업**
- `develop`으로 머지할 때 **PR(Pull Request)** 사용
- 공통 파일 수정이 필요하면 **담당자에게 요청** 후 머지

### 공통 파일 수정 권한

| 파일 | 수정 권한자 | 다른 사람이 변경 필요 시 |
|------|-----------|----------------------|
| `common/` 패키지 전체 | **A만** | A에게 Slack/이슈로 요청 |
| `components/` (공통 UI) | **E만** | E에게 요청 |
| `styles/variables.css` | **E만** | E에게 요청 |
| `layout.tsx` | **E 주도** | A가 인증 부분만 PR 제출 |
| `package.json` (FE) | **E 주도** | 라이브러리 추가 시 E를 통해 |
| `build.gradle` (BE) | **A 주도** | 의존성 추가 시 A를 통해 |

### 모듈 간 인터페이스 약속

1. **Port 인터페이스를 먼저 정의** (Sprint 0에서 합의)
2. 예: B가 `UserQueryPort`를 `event/` 모듈에 정의 → A가 `user/` 모듈에서 해당 구현체 제공
3. 예: F가 `LoadCommunityPort`를 통해 D의 커뮤니티 데이터 참조 → 인터페이스만 공유, 구현체는 각자 모듈
4. `internal/` 어댑터에서 처리하므로 실제 파일은 각자 모듈에만 존재
