커뮤니티 -장혜은🎤 발표 스크립트 

1. 배경 및 문제 의식 (Why)
"저희가 기획한 커뮤니티의 핵심은'이벤트가 끝난 뒤에도 이어지는 네트워킹'입니다. 단순히 이벤트 참여자만 모이는 닫힌 공간이 아니라, 관심 있는 누구나 정보를 볼 수 있게 열어두어 플랫폼 유입을 유도하고자 했습니다.

하지만 여기서 고민이 생겼습니다. 누구나 글을 쓸 수 있다면 기존의 다른 커뮤니티와 차별점이 없었습니다. 그래서 저희는 조회는 모두에게 열려있지만, 참여(작성)는 자격이 검증된 유저만 가능한 구조를 선택했습니다."

2.
"가장 까다로웠던 점은 권한 검증의 복잡도였습니다. 단순히 로그인 여부만 체크하는 게 아니라, 이벤트를 구매했는지(Order), 뱃지를 획득했는지(Badge) 등 서로 다른 도메인의 데이터를 통합해서 판단해야 했기 때문입니다."

특히 커뮤니티 성격에 따라 검증 로직이 달라졌습니다.

자동생성: 실 결제 완료 여부(PAID/REGISTERED) 확인

유저 생성형: 특정 이벤트 뱃지 보유 여부 확인

이처럼 매번 데이터 포인트가 달라지는 구조에서 코드의 파편화를 막아야 했습니다.

3. 해결책: CommunityPermissionService (How)
이를 해결하기 위해 권한 검증 로직을 전담하는 CommunityPermissionService를 구축했습니다.

게시글을 쓸 때나 페이지에 진입할 때 등, 권한이 필요한 모든 시점에서 이 서비스를 호출하도록 구조를 단일화했습니다. 덕분에 여러 도메인(주문, 뱃지, 회원)의 복잡한 데이터를 한곳에서 깔끔하게 판단할 수 있게 되었고, 비즈니스 로직과 권한 검증 로직을 명확히 분리할 수 있었습니다.

4. 성과 및 확장성 (Result)
결과적으로 두 가지 이점을 얻었습니다.

보안성: 상태 기반 검증을 통해 단순 구매 이력이 아닌, 현재 '유효한 참여자'인지를 실시간으로 체크합니다.

확장성: 향후 구독제나 레벨 시스템 같은 새로운 커뮤니티 타입이 추가되더라도, 기존 로직을 건드리지 않고 CommunityType 전략만 추가하면 되는 구조를 완성했습니다."

//상태 기반 검증: 단순 구매 이력이 아니라, 현재 주문이 '결제 완료' 상태인지 실시간으로 체크하여 보안성을 높였습니다.

//유연한 확장성: 나중에 '구독자 전용'이나 '레벨 기반' 커뮤니티가 추가되더라도, 기존 코드를 크게 수정하지 않고 검증 조건만 추가하면 되는 구조를 갖췄습니다."

* 어떤 이벤트에 등록하고 캘린더를 열어서 등록하는 절차가 귀찮았음. 그래서 결제완료 메일에 캘린더를 연동해서 링크를 클릭하면 저장이 될 수 있도록 구글 캘린더의 슬러그 기반 저장 기능을 활용해보았다. 추후 마이페이지 등에 링크로 추가하려고 한다.
* 호스트가 티켓을 어떤 식으로 생성하느냐에 따라 참여자가 자칫 중복된 세션을 구매할 수도 있음. 그래서 장바구니에 같은 세미나의 티켓을 이미 갖고 있다면 안내 메세지를 띄우는 안전장치를 마련함

마이페이지 - 박단희
1. 마이페이지 대시보드 바로가기 연동
이벤트 탭 부분에서는 단순히 DB에 저장된 상태를 불러오는 데 그치지 않고, 이벤트의 시작/종료 일시 데이터와 시스템의 **현재 시간(Real-time)을 비교 검증(Validation)**하는 로직을 추가했습니다. 이를 통해 별도의 수동 상태 업데이트 배치 작업 없이, 진행 전, 진행 중, 종료 탭으로 이벤트 리스트가 비즈니스 로직에 의해 동적으로 자동 필터링되도록 설계하여 사용자에게 항상 최신의 이벤트 상태를 적시에 제공


"마이페이지에 딱 들어오면 가장 먼저 보이는 요약 박스들을 버튼처럼 만들었습니다. 단순히 '몇 개 있는지'만 보여주는 게 아니라, 박스를 클릭하면 즉시 해당 탭이 열리도록 만들었어요. 주소창의 꼬리표(파라미터)를 읽어와서, 탭을 알아서 척척 열어주게끔 구현해서 편의성을 높였습니다."

2. 이벤트 목록 자동 분류 기능
이벤트의 시작/종료 일시 데이터와 시스템의 **현재 시간(Real-time)을 비교 검증(Validation)**하는 로직을 추가했습니다. 이를 통해 별도의 수동 상태 업데이트 배치 작업 없이, 진행 전, 진행 중, 종료 탭으로 이벤트 리스트가 비즈니스 로직에 의해 동적으로 자동 필터링되도록 설계


"이벤트 탭에서는 사용자가 참여한 이벤트를 '진행 전', '진행 중', '종료'로 나누어 보여줍니다. 이건 수동으로 바꾸는 게 아니라, 서버시간과 행사 일정을 비교해서 날짜가 지나가면 알아서 '종료' 탭으로 넘어가도록 자동으로 분류되게 만들었습니다."

3. 환불 처리 절차

결제 내역(/orders) 내 환불 기능의 경우, 일반적인 C2C 모델을 참고해 단순 취소가 아닌 **호스트 검토 및 승인 파이프라인(Workflow)**으로 쪼개서 작업 → 유저가 마이페이지 상세 내역에서 환불을 요청하면, 해당 요청 상태가 트랜잭션 관리 되어 주최자에게 전송되고, 주최자 어드민에서 이벤트를 트리거하여 최종적으로 승인/반려 사이클이 완료되도록 권한 기반 비즈니스 로직을 구현

"결제한 내역을 환불할 때 그냥 취소되는 게 아니라 실제 서비스처럼 절차를 만들었습니다. 사용자가 마이페이지에서 환불을 신청하면, 주최자에게 그 내역이 전달됩니다. 그럼 주최자가 확인 후 승인해야 최종적으로 환불이 완료되는 안전한 구조로 만들었습니다."

4. 찜 목록 (Wishlist)
유저들이 사이트 전반에서 경험한 데이터를 파편화시키지 않도록, 찜 목록(Wishlist)과 커뮤니티 활동(Community) 데이터를 마이페이지에서 중앙 집중화(Aggregation) → 메인 페이지 전역에서 발생한 찜 이벤트가 위시리스트 컴포넌트에 즉각 반영되고, 자신이 활동 중인 커뮤니티 목록과 게시물 북마크를 한곳에서 조회할 수 있도록 API 통신 최적화와 레이아웃 구조를 고려



"메인 페이지에서 하트(찜)를 누른 강의들은 마이페이지 첫 번째 탭에서 한눈에 모아볼 수 있도록 했습니다. 사이트 여기저기 흩어진 관심사들을 딱 이 공간에 모아주었다고 생각하시면 됩니다."
"마이페이지 내 커뮤니티 탭에서는, 내가 속해 있는 커뮤니티 목록뿐만 아니라 제가 특별히 북마크(저장) 해둔 게시물들만 모아서 볼 수 있는 기능도 추가하여 커뮤니티 활동을 더 편하게 했습니다."

6. 홀로그램 뱃지 (확장 비전)

수강을 완료하면 확인증처럼 멋진 홀로그램 효과가 들어간 뱃지를 받게 되는데요, 나중에는 링크드인처럼 유저들끼리 뱃지 여부를 공개하게 만들 예정입니다. 다른 사람의 프로필을 보고 '오, 이 사람 이런 강의 듣고 커리어를 쌓았네? 나도 들어봐야지!' 하면서 서로 자극을 받고 활동을 이끌어내는, 이른바 성장형 커뮤니티 플랫폼이 될 수 있도록 공개 여부 설정 기능까지 미리 고려해 두었습니다.

추후 이 기능은 초기 단계의 링크드인처럼, 유저들이 서로의 배지를 통해 각자의 커리어 빌딩 과정을 공유하는 소셜 기능으로 확장할 예정입니다. '이 사람은 이 강의를 듣고 이렇게 커리어를 키워나갔구나'라는 영감을 주며, 완성된 결과보다 성장 과정을 함께하는 커뮤니티 생태계를 목표로 이를 대비한 '배지 공개 여부 필드(Visibility)' 구조까지 선제적으로 설계해두었습니다."

7. 프로필 변경 동기화 및 회원 탈퇴 트러블슈팅

회원 탈퇴 기능 구현 시 처음에는 안전한 데이터 관리를 위해 Soft Delete(is_deleted=true) 방식으로 구현, 그러나 탈퇴 후 **동일한 이메일로 재가입을 시도할 때, Unique Constraints(고유성 제약조건) 오류나 '이미 존재하는 회원'이라는 예외가 터지는 사이드 이펙트(Side Effect)**가 발생 -> 이를 해결하기 위해 현업의 개인정보 보호법(GDPR 준수)에 해당하는 파기관리를 고려해 **Hard Delete 방식(물리적 레코드 삭제)**으로 비즈니스 로직을 변경했고, 더불어 데이터 정합성을 위해 외래키(Foreign Key) 계단식 삭제(Cascade) 처리까지 무결점하게 업데이트해 이슈를 해결

"기능을 만들면서 겪은 문제(트러블 슈팅)도 말씀드리고 싶습니다. 처음 탈퇴 처리를 할 때는 그냥 '탈퇴한 회원입니다(Soft Delete)'라고 표시만 해뒀는데요, 이러니까 같은 이메일로 다시 가입하려고 할 때 '이미 있는 회원입니다'라는 에러가 났습니다. 그래서 개인정보 보호도 확실히 할 겸 탈퇴 시 DB에서 깔끔하게 기록을 삭제(Hard Delete)하도록 구조를 고쳐서 에러를 해결했습니다. 그리고 프로필 사진 변경하면 관리자 모드, 참여자 명단 어디서든 바로 갱신되도록 상태 관리도 꼼꼼하게 처리했습니다. "

8. 소셜 로그인과 회원정보(비밀번호) 탭 분리
유저가 프로필 이미지를 변경했을 때, 마이페이지뿐만 아니라 Admin, Host 대시보드 등 전역적인 컴포넌트에서 수정 사항이 실시간으로 동기화 → 구글 소셜 로그인의 경우 해당 OAuth Provider(Google)가 인증을 위임받기 때문에 내부적인 패스워드가 무의미합니다. 플랫폼 보안성에 혼동을 주지 않기 위해 유저의 인증 공급자(Provider) 타입을 API를 통해 판별했습니다. provider === 'LOCAL' 일 때만 /security 탭을 조건부 렌더링(Conditional Rendering)하여 네이티브 가입자만 비밀번호 수정 페이지에 접근할 수 있도록 UI와 클라이언트 권한 제어를 분리해 구현


"구글로 로그인하신 분들은 굳이 우리 사이트에서 비밀번호를 바꿀 필요가 없잖아요? 그래서 계정을 검사해서 일반 가입자일 때만 '계정 보안(비밀번호 변경)' 메뉴가 보이도록 아주 자연스럽게 메뉴를 숨겨주는 로직도 넣었습니다."

9. 1:1 문의 진행 상태 실시간 안내
1:1 문의 게시판을 단순한 CRUD가 아닌 상태 머신(State Machine) 기반의 라이프사이클로 고도화했습니다. 유저가 문의를 작성하면 PENDING(대기 중) 상태가 되며, Admin이 패널에서 데이터를 조회하는 즉시 IN_PROGRESS(검토 중)로 상태를 Transition(전환) 시킵니다. 검토 중 이전까지만 유저가 자진 취소(Soft Cancel)가 가능하게 하여 관리자의 데드락을 없앴고, 최종적으로 승인(처리 완료) 혹은 반려 되며 반려 시 Admin이 남긴 사유를 Mapping해 고객 관리를 위한 CS 루프를 완성


"1:1 문의 기능의 전체적인 흐름을 제가 짰는데요. 단순히 글 하나 남기는 게 아닙니다. 사용자가 남기면 '대기 중' 👉 관리자가 글을 읽기만 해도 '검토 중'(이때까진 사용자가 취소 가능) 👉 마침내 해결되면 '처리 완료' 👉 안타깝게 거절되면 **'반려'**라는 상태로 바뀌게 꼼꼼히 설계했습니다. 반려된 경우 관리자가 남긴 이유도 볼 수 있어서 답답함을 없앴습니다."

10.숨겨둔 로그아웃

"마지막으로, 로그아웃 버튼을 첫 화면 헤더에 두지 않고 회원정보 밑바닥에 배치했습니다. 단순히 숨긴 게 아니라 최근 인스타그램이나 유튜브처럼 의도적인 디자인을 따른 겁니다. 사용자가 툭 치고 실수로 로그아웃 되어서 흐름이 끊기는 걸 막기 위한 장치이죠. 한 번 로그인하면 최대한 사이트에 길게 머물게 하려는 UX(사용자 경험) 디테일까지 신경 썼습니다."
