# 🎤 VenueOn — 서비스 발표안

> **"이벤트는 끝나도, 연결은 계속된다."**

---

## 1. 우리가 해결하려는 문제

### 💬 우리 팀의 실제 경험에서 시작합니다

저희 팀은 지금 뉴렉처 교육과정을 수강하고 있습니다.
그런데 돌이켜보면, **가장 많이 배운 순간**은 강의를 듣는 시간이 아니었습니다.

> *"강사님의 수업을 일방향으로 들었을 때보다,
> 슬랙에서 서로 이야기하며 지식과 관심사를 공유했을 때
> 훨씬 더 좋은 효과가 있었습니다."*

같은 주제를 공부하는 사람들끼리 **"이거 어떻게 풀었어?"**, **"이 기술 써봤는데 괜찮더라"** 하고 주고받는 그 대화들—  
그게 진짜 성장의 순간이었습니다.

그런데 한 가지 의문이 들었습니다.

> *"이 교육이 끝나면, 이 관계도 끝나는 건가?"*

이 질문이 VenueOn의 출발점이었습니다.

### 🔴 기존 교육 플랫폼을 돌아보니 — "일방통행의 끝"

우리의 경험을 되짚어보며, 기존 교육 플랫폼들을 살펴봤습니다.
결론은 동일했습니다. **어디에도 "이후의 연결"은 없었습니다.**

#### 1-1. 지식 공유자와의 소통 단절

| 플랫폼 | 문제점 |
|--------|--------|
| **인프런** | 질문을 올려도 즉각적인 답변을 기대하기 어려움. '지식공유자 답변' 기능이 있지만, 실시간 소통은 불가능하고, 지식공유자가 답변 기능 자체를 제공하지 않을 수도 있음 |
| **클래스101** | 온라인 강의를 들으면서 옆자리 친구와 이야기하며 같이 성장하는 경험을 할 수 없음. 오프라인 수업의 그 활기가 사라짐 |
| **Udemy / Coursera** | 글로벌 플랫폼이지만 커뮤니티는 포럼 수준. 같은 강의를 듣는 한국어 사용자끼리 연결되기 어려움 |

#### 1-2. "강의가 끝나면, 관계도 끝난다"

이것이 모든 플랫폼의 공통적인 구조적 한계입니다:

```
[수강 신청] → [강의 시청] → [수료] → … 그리고 끝.
```

- 수강이 끝나면 같은 주제에 관심 있는 사람들과의 접점이 **완전히 사라짐**
- 강의에서 배운 내용을 나누고 싶어도 **채널이 없음**
- 결국 사용자는 플랫폼을 **콘텐츠 자판기**처럼 사용하고 떠남

우리가 슬랙에서 느꼈던 그 경험—**같은 걸 배우는 사람들끼리의 자유로운 소통**—을
플랫폼 안에 녹여낼 수는 없을까?

## 2. 우리의 해답 — VenueOn

### 💡 핵심 컨셉: "이벤트 × 커뮤니티" 플랫폼

VenueOn은 단순한 강의 판매 플랫폼이 아닙니다.
**이벤트를 매개로 사람들이 모이고, 커뮤니티를 통해 계속 연결되는 플랫폼**입니다.

```
기존 플랫폼:   [콘텐츠] → [소비] → [이탈]
VenueOn:      [이벤트] → [참여] → [커뮤니티] → [재방문] → [새 이벤트 발견] ↺
```

### 2-0. "슬랙이나 디스코드로 충분하지 않나요?"

이 질문은 반드시 나올 수 있습니다. 답은 **"근본적으로 다르다"**입니다.

| | 슬랙 / 디스코드 / 카카오 오픈채팅 | VenueOn |
|---|---|---|
| **대상** | 특정 집단(회사, 스터디) 중심 | 불특정 다수가 주제로 모이는 구조 |
| **형성 계기** | 누군가 초대해야 시작 | 이벤트 참여 → 자동으로 커뮤니티 입장 |
| **플랫폼 성장** | 메신저는 메신저. 사업과 무관 | 커뮤니티가 커지면 **플랫폼도 같이 성장** |
| **비즈니스 연결** | 없음 | 호스트가 커뮤니티 안에서 **다음 이벤트를 자연스럽게 홍보** |

핵심은 VenueOn에는 **사용자만 있는 게 아니라 이벤트를 만드는 호스트도 있다**는 것입니다.
커뮤니티가 활성화되면 호스트에게는 홍보 채널이 되고, 사용자에게는 다음 이벤트의 발견 경로가 됩니다.
이 선순환이 **슬랙에서는 절대 만들어지지 않는 비즈니스 가치**입니다.

### 2-1. 이벤트가 끝난 뒤에도 이어지는 네트워킹

| 기존 플랫폼 | VenueOn |
|------------|---------|
| 강의 종료 → 관계 끝 | 이벤트 종료 → 커뮤니티 시작 |
| 질문은 Q&A 게시판에만 | 가벼운 스몰토킹부터 전문 토론까지 |
| 지식공유자 답변 의무 없음 | 같은 관심사의 참여자끼리 자유롭게 소통 |

### 2-2. "꼭 전문적인 질문만 해야 할 것 같다"를 깨다

인프런에서 질문을 올릴 때, 이런 생각이 들지 않았나요?

> *"이 정도 질문을 올려도 되나…?"*
> *"너무 기초적인 건 아닌가…?"*

VenueOn의 커뮤니티는 **진입 장벽이 없습니다.**

- 🎯 같은 주제로 **진지한 지식 공유**를 할 수도 있고
- 😄 같은 주제로 **재밌는 잡담과 농담따먹기**를 할 수도 있는
- 🌐 **자유로운 분위기의 커뮤니티**를 지향합니다

핵심은 **"같은 이벤트에 참여했다"는 공통 경험**이 자연스러운 소속감을 만든다는 것입니다.

## 3. 비즈니스 핵심 전략 — 리텐션 플라이휠

### 🔄 VenueOn의 성장 엔진

```
                    ┌─────────────────┐
                    │   이벤트 개설    │
                    │  (호스트가 생성)  │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │   참여자 유입    │
                    │  (티켓 구매)     │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │  커뮤니티 활성화  │
                    │ (수강 후 자동 생성)│
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ 콘텐츠    │  │ 뱃지      │  │ 신규      │
        │ 생산      │  │ 수집 욕구  │  │ 이벤트    │
        │ (게시글)  │  │ (성취감)  │  │ 발견      │
        └────┬─────┘  └────┬─────┘  └────┬─────┘
             │             │             │
             └──────────────┼──────────────┘
                            │
                            ▼
                   ┌─────────────────┐
                   │  플랫폼 잔류     │
                   │  (DAU/MAU 상승)  │
                   └────────┬────────┘
                            │
                            ▼
                   ┌─────────────────┐
                   │  네트워크 효과    │
                   │  (더 많은 호스트)  │
                   └─────────────────┘
```

이것이 VenueOn의 **리텐션 플라이휠(Retention Flywheel)** 입니다.

이벤트가 하나 열릴 때마다 커뮤니티가 하나 생기고,
커뮤니티가 활성화될수록 사용자는 플랫폼에 머물며,
활성 사용자가 많아질수록 새로운 호스트가 유입됩니다.

#### 🧊 콜드 스타트(Cold Start) 문제 — 어떻게 대응하는가?

> *"커뮤니티는 만들어놨는데 아무도 글을 안 쓰면?"*

VenueOn은 이 문제를 **2단계 전략**으로 해결합니다.

```
[1단계: 시드(Seed) 커뮤니티]
  이벤트가 생성되면 → 호스트가 매니저인 커뮤니티 자동 생성
  → 호스트가 첫 콘텐츠를 올리고, 수강생과 직접 소통
  → "빈 방에 들어가는 게 아니라, 호스트가 이미 있는 방에 합류"

[2단계: 자생적 파생]
  호스트 중심 커뮤니티가 성장하면
  → 해당 주제에 대해 다른 방향의 커뮤니티가 자연 발생
  → 예: "Python 세미나" 커뮤니티 → "Python 취업 스터디" 커뮤니티 파생
```

핵심은 **처음부터 빈 커뮤니티를 던져주는 게 아니라, 호스트라는 리더가 있는 상태에서 시작한다**는 것입니다.

### 3-1. 뱃지 — 수집 욕구를 자극하는 성장형 소셜 모델

```
[이벤트 결제 완료] + [수강 완료 확인]
        │
        ▼
[홀로그램 뱃지 발급] → "나 이 분야 공부했어" 증명
        │
        ▼
[뱃지 보유자 전용 커뮤니티 입장 자격 획득]
        │
        ▼
[프로필에 뱃지 공개] → 링크드인처럼 전문성 어필
        │
        ▼
[더 많은 이벤트 참여 동기 부여] ↺
```

- 뱃지는 단순 장식이 아닌 **커뮤니티 입장권**
- 뱃지가 쌓일수록 프로필의 **전문성이 시각적으로 축적**
- 뱃지 공개/비공개를 사용자가 직접 선택 (프라이버시 존중)
- ⚠️ **뱃지 발급 조건은 이중 검증**: 결제 완료 + 수강 완료 시점에만 발급되므로, 결제만 하고 참석하지 않은 사용자에게는 뱃지가 발급되지 않음 → 커뮤니티의 신뢰성 유지

### 3-2. 커뮤니티 권한 체계 — 자격이 만드는 신뢰

VenueOn 커뮤니티의 차별점은 **"누구나 볼 수 있지만, 아무나 쓸 수 없다"** 는 것입니다.

| 유형 | 입장 조건 | 예시 |
|------|----------|------|
| 자동 생성형 | 해당 이벤트 결제 완료 | "Python 풀스택 세미나" 수강생 전용 |
| 뱃지 보유형 | 특정 뱃지 보유 | "AWS 자격증 스터디" 뱃지 보유자 전용 |

조회는 누구에게나 열려있어 **탐색 욕구**를 자극하고,
작성 권한은 자격 검증을 통해 부여하여 **커뮤니티의 질**을 보장합니다.

## 4. 차별화 포인트 — "이벤트 기반" 아키텍처의 확장성

### 🎯 왜 "강의"가 아니라 "이벤트"인가?

이것이 VenueOn의 **가장 중요한 설계 철학**입니다.

지금은 강의와 세미나 형식에 초점을 맞추어 개발했지만,
VenueOn은 처음부터 **"이벤트"를 기본 단위**로 설계했습니다.

```
"이벤트"라는 추상화 레이어 위에 무엇이든 올릴 수 있다
```

### 4-1. 수평 확장 시나리오

| Phase | 이벤트 유형 | 예시 |
|-------|-----------|------|
| **현재** | 강의 · 세미나 | IT 개발 부트캠프, 디자인 워크숍 |
| **Phase 2** | 컨퍼런스 · 밋업 | 개발자 컨퍼런스, 스타트업 네트워킹 |
| **Phase 3** | 문화 · 체험 | 와인 테이스팅, 독서 모임, 쿠킹 클래스 |
| **Phase 4** | 스포츠 · 레저 | 러닝 크루, 등산 모임, 요가 리트릿 |
| **Phase 5** | 비즈니스 | 투자 설명회, 채용 박람회, 전시회 |

핵심: **코어 도메인(Event-Session-Ticket)의 변경 없이** 새로운 카테고리만 추가하면 됩니다.

### 4-2. 수직 확장 시나리오 — 커뮤니티 네트워크 효과

```
Phase 1: 이벤트 1개 → 커뮤니티 1개 → 사용자 30명
Phase 2: 이벤트 10개 → 커뮤니티 10개 → 사용자 300명
Phase 3: 이벤트 100개 → 커뮤니티가 서로 교차 → 사용자 간 다중 소속
         → "Python 커뮤니티" 사용자가 "데이터 분석 커뮤니티"도 발견
         → 자연스러운 크로스셀링(Cross-selling)
```

**확장할수록 커뮤니티가 커지고, 커뮤니티가 커질수록 사용자를 더 오래 묶어둘 수 있습니다.**

## 5. 사용자 여정 (User Journey)

### 👤 참여자의 여정

```
1️⃣ 발견
   "흥미로운 세미나를 발견했다"
         │
         ▼
2️⃣ 탐색
   이벤트 상세 페이지에서 세션 구성, 강사 정보, 잔여석 확인
   커뮤니티 미리보기 → "수강하면 이런 커뮤니티에 참여할 수 있구나"
         │
         ▼
3️⃣ 결제
   원하는 세션만 골라 티켓 구매 (유연한 패키징)
   Google Calendar 자동 연동 링크 제공
         │
         ▼
4️⃣ 참여
   이벤트 수강 · 뱃지 획득
         │
         ▼
5️⃣ 커뮤니티 진입
   "이 주제 재밌었는데, 더 이야기하고 싶다"
   → 자격 검증 후 커뮤니티 자동 입장
         │
         ▼
6️⃣ 재방문 루프
   커뮤니티 활동 → 새로운 이벤트 발견 → 재구매
```

### 🎙️ 호스트의 여정

```
1️⃣ 호스트 등록
   프로필 작성 · 전문 분야 설정
         │
         ▼
2️⃣ 이벤트 설계
   세션 구성 → 티켓 패키징 (ALL-IN-ONE / 개별 / 묶음 등)
   → Smart Default로 복잡한 설정 자동화
         │
         ▼
3️⃣ 운영
   대시보드에서 이벤트 현황 · 수강생 · 주문 한눈에 파악
   실시간 상태 뱃지로 판매 현황 모니터링
         │
         ▼
4️⃣ 사후 관리
   커뮤니티를 통해 수강생과 지속적 관계 유지
   → 다음 이벤트의 자연스러운 홍보 채널 확보
```

## 6. 기술 아키텍처 — 왜 이렇게 만들었는가

### 🏗️ 기술 스택 한눈에 보기

| 영역 | 선택 | 이유 |
|------|------|------|
| **프론트엔드** | Next.js 16 (App Router) | SSR/SSG + Server Component로 SEO와 성능 확보 |
| **스타일링** | Vanilla CSS Module | 컴포넌트별 스코프 격리, 외부 의존 없음 |
| **백엔드** | Spring Boot 3.x + Java 17 | 엔터프라이즈급 안정성, 풍부한 생태계 |
| **아키텍처** | 헥사고날 (Ports & Adapters) | 모듈 간 독립성 → 팀 협업 최적화 |
| **DB** | PostgreSQL 15 (27개 테이블) | 트랜잭션 안정성 + JSON/Full-text 확장성 |
| **캐시** | Redis 7 | JWT 블랙리스트, 세션 캐시 |
| **결제** | Toss Payments (테스트 모드) | 국내 최적화 결제 API |
| **인증** | JWT + iron-session (BFF) | 보안 강화 + SSR 호환 |

### 6-1. "헥사고날 아키텍처 — 5명에게 오버 엔지니어링 아니었나?"

결론부터 말하면, **5명이라서 오히려 헥사고날이 효과적이었습니다.**

기존 3-Layered 구조에서는 `Service` 한 파일에 여러 기능이 몰리면서
팀원이 동시에 같은 파일을 수정하는 **머지 충돌(Merge Conflict)**이 빈번합니다.

헥사고날에서는 각 기능이 UseCase → Port → Adapter로 명확히 분리되어 있어,
**다른 기능을 개발하는 팀원끼리 같은 파일을 건드리는 상황이 극히 드물었습니다.**

```
기존 (3-Layered):
  EventService.java ← 팀원 A, B, C가 동시에 수정 → 💥 충돌

헥사고날 (Ports & Adapters):
  GetEventListUseCase.java    ← 팀원 A만 작업
  CreateEventUseCase.java     ← 팀원 B만 작업
  UpdateEventStatusUseCase.java ← 팀원 C만 작업
  → 🟢 충돌 거의 없음
```

실제로 13개 도메인 모듈과 Actor 계층(host/admin/manager)을 분리한 결과,
**기능 단위의 병렬 개발**이 가능해져 5명의 생산성을 극대화할 수 있었습니다.

### 6-2. "이벤트 상태를 DB에 안 넣으면 성능은?"

VenueOn의 이벤트 상태(ONGOING, 모집중 등)는 **DB에 저장하지 않고 조회 시 계산(Computed)**합니다.

> *"이벤트가 수만 개가 되면 조회 때마다 세션의 시간을 계산해야 하니 느려지지 않나?"*

이 트레이드오프를 인지하고 있으며, 다음과 같이 대응했습니다:

| 대안 | 문제점 | VenueOn의 선택 |
|------|--------|----------------|
| DB 저장 + 스케줄러 | 스케줄러가 **상시 구동**되며 전체 이벤트를 순회 → 오히려 부하 증가 | ❌ 채택하지 않음 |
| **Computed + 페이지네이션** | 한 페이지에 12~20개만 조회 → 계산 대상 극히 적음 | ✅ 채택 |

현실적으로 사용자는:
- 수천 개의 이벤트를 한 번에 보지 않고, **필터링 + 페이지네이션**으로 탐색
- 리스트 페이지에 오래 머무르지 않으므로, 조회 시점 계산으로 충분히 실시간에 가까운 상태를 보여줄 수 있음
- 스케줄러를 돌리는 편이 오히려 **사이트 전체 부하가 더 크다**고 판단

### 6-3. "왜 프론트에서 JWT를 직접 백엔드로 안 보내고 BFF를 거치나?"

VenueOn은 **iron-session + Next.js Route Handler**로 JWT 프록시(BFF) 패턴을 사용합니다.

```
일반적인 방식 (위험):
  브라우저 → localStorage에 JWT 저장 → 백엔드 직접 호출
  ⚠️ XSS 공격 시 JWT 탈취 가능

VenueOn의 방식 (BFF):
  브라우저 → 암호화된 httpOnly 쿠키 → Next.js 서버 → 백엔드
  🔒 JavaScript에서 토큰 접근 불가
```

이 구조의 핵심 이점 3가지:

| # | 이점 | 설명 |
|---|------|------|
| 1 | **XSS 방어** | JWT가 httpOnly 암호화 쿠키에 저장되어, 브라우저 JavaScript에서 접근 불가 |
| 2 | **SSR 호환** | Next.js Server Component에서도 인증된 API 호출 가능 (쿠키 자동 전달) |
| 3 | **API 은닉** | 백엔드 URL이 브라우저에 노출되지 않음 → 공격 표면 최소화 |
